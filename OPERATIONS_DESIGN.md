# CDP免許更新管理 業務・データ設計

## 1. 設計方針

このシステムは、独自の業務解釈を作らず、一般的な「マスタ、案件、証憑、請求、入金、監査ログ」の構造で管理する。

- 画面表示用の計算結果を原本として保存しない。原本日付と取引明細から毎回再計算する。
- 対象者、更新案件、請求、入金を1行に詰め込まない。
- 金額を入力しただけでは売上・請求済・未収金にしない。
- 削除は原則行わず、取消・無効化と訂正履歴を残す。
- CSV取込は「検証、プレビュー、確定」の3段階とし、一部行だけを黙って取り込まない。
- 本番の正本は共有データストアとし、ブラウザ `localStorage` は正本にしない。

## 2. 正本データ

本番では次の表を分ける。各表は変更されない内部IDを主キーにする。

### persons（対象者マスタ）

`person_id`, 氏名, 会社ID, 電話, メール, 有効/無効, 作成日時, 更新日時

氏名は表示項目であり、同姓同名判定や更新キーには使わない。

### licenses（技能証明マスタ）

`license_id`, `person_id`, 技能証明書番号, 一等/二等, 航空機の種類, 交付日, 有効期限, 確認元, 確認日, 証憑ID

有効期限は技能証明書またはDIPSで確認した日付を保存する。計算で新しい有効期限を確定しない。

### renewal_cases（更新案件）

`case_id`, `person_id`, 対象技能証明ID, 管理年度, 受講回数, 受講機関, 申込日, 講習予定日, 講習修了日, 受講開始時照合結果・照合日・照合者・証憑ID, DIPS修了者情報連携日, 本人DIPS更新申請日, 更新完了確認日, 新技能証明ID, 状態, 担当者

1回の更新を1案件とする。次回更新は同じ行を上書きせず、新しい案件を作る。

### certificates（講習・身体適性資料）

`certificate_id`, `case_id`, 種別, 証明書番号, 発行日/確認日, 実交付日, 有効期限, 発行者, 証憑ID

講習修了証明と身体適性資料を別明細にする。証明書再発行でも当初の有効期限を勝手に延長しない。

### documents（証憑）

`document_id`, 所有表種別, 所有ID, 文書種別, DriveファイルID, ファイル名, 取得日, 確認者, ハッシュまたは版

個人情報を一覧画面へ常時出さない。閲覧権限は業務上必要な担当者に限定する。

### artifact_outputs（成果物生成台帳）

`output_id`, `case_id` または対象レコードID, 成果物種別, 入力ハッシュ, 版, 状態, DriveファイルID, ファイル名, 出力先フォルダID, 作成日時, 作成者, 採番情報, エラー内容

成果物の生成事実と、証明書の交付、DIPS修了者情報連携、本人のDIPS更新申請、見積送付、請求発行は分ける。同一入力ハッシュの再実行では既存成果物を参照し、入力変更時は旧版を消さずに新しい版を作る。作成失敗も監査対象として残す。

### invoices / invoice_lines（請求書・請求明細）

請求書: `invoice_id`, 請求書番号, 請求先ID, 取引日, 発行日, 支払期限, 状態（下書き/発行済/取消）, 端数処理方法

明細: `invoice_line_id`, `invoice_id`, `case_id`, 品目, 数量, 単価税抜, 値引税抜, 税区分, 税率

請求書番号は発行時に採番し重複不可とする。下書きは債権集計に含めない。取消は元請求書を消さず、理由と訂正先を残す。

### payments / payment_allocations（入金・消込）

入金: `payment_id`, 入金日, 金額, 方法, 振込名義, 参照番号

消込: `allocation_id`, `payment_id`, `invoice_id`, 消込額

分割入金、複数請求の一括入金、過入金、返金を扱うため、入金を請求書の1項目として上書きしない。

### audit_logs（操作履歴）

`log_id`, 操作日時, 操作者, 操作, 対象表, 対象ID, 変更前, 変更後, 取込バッチID

本番ログは件数で切り捨てない。講習記録・帳簿は講習事務終了後3年間を下回らないよう保持し、削除権限、廃棄承認、廃棄ログを運用規程で定める。

## 3. 期限の計算

日付の正本は技能証明書、DIPS、証明書記載値である。システムの計算日は案内・確認用とし、法的な確定値として上書きしない。

```text
DIPS更新可能開始日 = 現在の技能証明有効期限の6か月前
社内申請目標日     = 現在の技能証明有効期限の1か月前
講習修了証明期限   = 修了証明書初回発行日の3暦月後の前日
DIPS機関連携期限   = 修了証明書発行後5営業日以内（起算日・閉庁日は要確認）
対応期限           = 上記のうち、その案件に適用される最も早い日
```

修了証明書期限は、3暦月後に同じ日があればその前日、同じ日がなければ対象月の末日とする。例: 6月10日発行は9月9日まで、1月31日発行は4月30日まで。ブラウザの時刻差を避けるため、業務日付は `YYYY-MM-DD` として扱う。

国土交通省の公開案内では、技能証明の有効期限は3年、更新は満了日の6か月前から可能で、1か月前までの本人申請が案内されている。作成時は修了証明書初回発行日から期限を計算し、初回発行日と実際の交付日は分けて記録する。訂正・再発行日で初回発行日を上書きして期限を延長しない。

DIPSの5営業日はCDPが行う修了者情報連携であり、本人申請期間とは別である。公開資料だけでは起算日・閉庁日が確定しないため、翌営業日を1日目とする値は暫定とし、担当部署への確認日・確認者・追加閉庁日を管理する。祝日マスタ未収録年または確認記録なしでは期限を推測しない。

身体適性資料の期限は資料に記載された期限を正本とし、CDPの確認日から自動計算しない。

## 4. 請求・消費税・未収金

### 請求額

```text
税率別税抜合計 = Σ(数量 × 税抜単価 - 税抜値引の絶対額)
税率別消費税   = 端数処理(税率別税抜合計 × 税率)
請求額税込     = Σ(税率別税抜合計 + 税率別消費税)
```

消費税の1円未満の端数処理は、1つの適格請求書につき税率ごとに1回だけ行う。明細ごとに端数処理して合計しない。端数方法は会社で切捨て・四捨五入・切上げのいずれかに固定し、請求ごとに担当者が自由変更しない運用を推奨する。

現在の成果物作成画面は単一料金明細として、`料金税抜 - 値引税抜の絶対額` を税抜対象額とし、その後に税率別の税を1回計算する。値引に負数を入力させて二重に加算しない。請求書には請求日と別に取引年月日（役務提供日）を表示する。作成した見積書・請求書は下書きであり、送付・発行を確認するまでは請求済額や未収金へ含めない。未発行・取消の税込計算額は参考値として債権計上額と別列にし、未知の税率・端数方式は推測計算せず0円計上・要確認とする。

更新講習等の通常の役務は10%として扱う。0%・8%を使う場合は、承認日、承認者、法令・税務判断の根拠を必須にする。非課税・不課税・対象外は将来0%と同一表示にせず、税区分を分ける。

### 集計

```text
請求済額 = 発行済かつ取消でない請求書の税込合計
実入金額 = 取消でない入金取引の合計
消込額   = 請求書へ割り当てた入金額
未収金   = max(0, 請求済額 - 消込額 - 貸倒/値引等の承認済調整額)
過入金   = max(0, 実入金額 - 消込額)
```

見積、下書き請求、料金未確定、取消請求を請求済額へ含めない。入金日と入金額はセットで管理し、返金は元入金を書き換えずマイナス取引または返金取引として記録する。

管理年度は4月1日から翌年3月31日とする。売上集計の期間は管理年度と混同せず、会社が採用する会計期間と取引日で抽出する。入金集計は入金日で切る。管理年度別の業務集計に請求額を表示する場合は参考値と明記し、会計集計へ流用しない。売上計上時期・勘定科目・仕訳は会計方針と税理士確認に従い、この更新管理画面だけで自動決定しない。

## 5. 取込・保存・閲覧・管理

### 取込

1. UTF-8 BOM付きCSVテンプレートを配布する。
2. 必須列、日付、数値、選択値、重複キー、参照先IDを全行検証する。
3. 追加・更新・無視・エラーの件数と先頭行をプレビューする。
4. エラーがある場合はバッチ全体を確定しない。
5. 確定前バックアップと取込バッチIDを保存する。
6. 確定後に件数照合し、監査ログを残す。

### 保存

- 本番保存はサーバー側APIで行い、更新日時または版番号による楽観ロックで同時更新を検出する。
- 複数表更新は可能な範囲で一括確定し、途中失敗を残さない。
- 自動バックアップに加え、復元テストを定期実施する。
- CSVとJSON出力はバックアップ補助であり、正式な権限管理・履歴管理の代わりにしない。

### 閲覧

- 一覧: 対応期限、必要対応、状態、担当者だけを優先表示する。
- 詳細: 個人情報、証憑、期限根拠、請求、入金、変更履歴をタブ分けする。
- 経理: 発行済請求、入金、未収、期限超過、過入金を別集計する。
- 集計: 対象期間、抽出条件、更新時刻、件数定義を画面とCSVに表示する。

### 権限

最低でも「閲覧」「更新担当」「経理」「管理者」を分ける。個人情報、請求取消、バックアップ復元、CSV一括取込、権限変更は個別に制御する。

## 6. 成果物の作成・保存・訂正

### 作成単位

対象者レコードをブラウザ内台帳へ保存した後、利用者が対象レコードと成果物種別を明示して作成する。バックグラウンド処理、日次一括処理、対象者保存と同時の自動作成は行わない。

作成できる成果物は、発行台帳、講習修了証明書、DIPS提出CSV、更新講習案内、講習記録簿、見積書・請求書の6種類とする。詳細は `ARTIFACT_AUTOMATION.md` を参照する。

### 作成前検査

1. 対象レコードID、対象者名、成果物別の必須項目を検査する。
2. 講習修了後の正式成果物は、技能証明申請者番号・氏名・住所・生年月日・DIPS通知メールの対象講習を照合した結果、日付、担当者、証憑参照を検査する。
3. 日付形式、証明書番号、DIPS申請者ID、税率、端数方式を検査する。
4. 講習記録簿は航空機の種類が回転翼航空機（マルチローター）であることと、各学科項目の日付、時刻、担当者、最低時間、同日時間帯の重複を検査する。実地は停止処分者向け講習が「あり」の場合だけ必須とし、「未確認」を「なし」へ推定しない。
5. 更新講習案内は4行の日程、会場、時間帯、受講可能期間内かを検査する。
6. 発行者、連絡先、適格請求書発行事業者登録番号、振込先、出力先を成果物ごとに検査する。
7. 無個人情報の専用原本、案内・記録簿原本の差込セルと構造、読取権限を確認する。
8. 明示設定された出力先が非公開で、Drive APIの全ページにあるuser/group権限が許可メール一覧内、domain/anyone権限なし、編集者による再共有不可であることを確認する。
9. DIPS CSVはCDP受講分で、営業日暦の確認日・確認者・追加閉庁日と対象年祝日が揃うことを確認する。
10. 空欄から証明書・DIPS番号を採番する場合は、既存台帳照合と採番正本への切替が完了していることを確認する。

検査は読取のみとし、Driveへの書込、採番予約、生成台帳への成功記録を行わない。警告とエラーを分け、エラーがある成果物は作成不可にする。

### 原本保護と出力先

- 既存のGoogleドキュメント、スプレッドシートは原本として読み取り、直接更新しない。実名タブ・既存データ入りファイルは全体コピーせず、無個人情報の専用原本を別途使用する。
- 原本を複製したコピー、または定義済みの新規ファイルだけへ記入する。
- 出力先は必須設定とし、その下に自動作成用フォルダを設け、対象レコード別に整理する。発行台帳と生成台帳は共通管理用として分ける。
- 自動作成用・対象レコード別フォルダは、親ID・対象レコードIDを含む固定識別マーカーで確認する。同名だけでは再利用せず、設定消失時も別用途フォルダへの誤帰属を防ぐ。
- 出力フォルダの共有範囲は必要最小限とし、リンク公開・ドメイン公開のフォルダには個人情報を作成しない。Drive APIでuser/group/domain/anyone権限とページング完了を検査し、許可一覧外・取得不能・編集者による再共有可なら停止する。
- 原本編集者を限定し、テンプレート変更は承認、清浄性検査、試作出力確認を経て反映する。

### 同時実行・冪等性・版

- `LockService` で採番、ファイル作成、生成台帳更新を直列化する。
- `対象レコードID | 成果物種別 | 入力ハッシュ` が一致する作成済みファイルは再作成せず、既存結果を返す。
- 内容が変わった場合は同じ対象レコード・種別の版を進め、旧版を残す。
- 訂正前後のファイルを上書き・削除せず、どの版を交付・送付・発行したかを業務記録で特定する。
- 複数成果物の作成は結果を種別ごとに返し、1件の失敗で作成済みの別成果物を隠さない。

### 状態を自動確定しない

ファイル作成だけでは、証明書交付済、DIPS修了者情報連携済、本人DIPS更新申請済、見積送付済、請求発行済、売上計上済にしない。実際の交付・連携・申請・送付・発行を確認した担当者、確認日、使用した成果物の版を別に記録する。

DIPSはCDP受講分のUTF-8 BOM付き11列CSVの作成までとし、ログイン、アップロード、申請は行わない。列7は固定値 `PA000000000000`、状態フラグは新規=`1`・訂正=`2`・削除=`3` とする。訂正・削除は誤登録時と同じ申請者IDを使い、自動採番しない。DIPSの認証情報は保存しない。最新のログイン後公式ひな形と列仕様は担当者が使用前に手動照合する。

## 7. 状態遷移

```text
受付前 → 案内対象 → 申込済 → 講習予定 → 講習修了
      → 証明書初回発行 → DIPS修了者情報連携 → 証明資料確認済
      → 本人DIPS更新申請済 → 更新完了確認済 → 次回管理登録済
```

状態は入力済み日付・証憑の組合せから判定する。担当者が一覧用ラベルだけを直接書き換えない。例外は「要確認」とし、理由、担当者、期限を必須にする。

## 8. 本番化の完了条件

- 接続先、所有者、バックアップ先、復元担当者が確定している。
- サンプルと本番データが完全に分離されている。
- 複数ユーザーの同時更新を検出できる。
- 請求書・入金・消込が別明細で保存される。
- 発行済請求だけで請求額と未収金が一致する。
- 消費税を請求書・税率ごとに1回計算するテストがある。
- 月末、うるう年、期限当日、取消、分割入金、過入金、再取込のテストがある。
- 操作ログから誰が何を変更したか追跡できる。
- 個人情報の閲覧権限とCSV出力権限が確認されている。
- 成果物の原本と出力コピーを区別でき、入力ハッシュ、版、作成結果を追跡できる。
- DIPS CSVを最新の公式ひな形と照合する担当者と確認日が決まっている。
- 対象者データの共有正本接続、メール、DIPS操作、トリガーは承認済みの接続先でレビュー後に有効化する。

## 9. 現在の運用範囲との差

現在の `Index.html` は、入力・閲覧・CSV・バックアップ・基本計算を行う単一ブラウザ試作であり、対象者データの共有正本には接続しない。一方、保存済みレコードを利用者が選び、作成前検査に合格して確定した場合に限り、6種類の成果物と生成台帳をDriveへ作成できる。

この成果物作成機能は、対象者データの共有保存、権限管理、複数ユーザー同時編集、複数入金、消込、返金、仕訳、正式な証憑保管を実現するものではない。したがって、ブラウザ内台帳を本番の共有台帳や正式会計帳簿として使用しない。
