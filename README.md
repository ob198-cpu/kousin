# CDP免許更新管理

更新対象者の入力・期限管理と、保存済みレコードから更新講習の成果物を作成する Google Apps Script アプリです。

Apps Script版には、専用の非公開スプレッドシートを対象者・権限の共有正本として使う保存API、ロール管理、楽観ロック、追記型監査、バックアップ、追記型イベントを正本とする正式会計台帳を実装しています。公開GitHub Pages版は、合成サンプルをメモリ上だけで表示する閲覧専用デモです。通常表示は旧 `localStorage` を読み込まず、対象者・監査・会計データを書き込みません。旧版で同じオリジンに保存されたデータは、利用者が「旧データをJSON退避」を明示した場合だけ救出用に読み取れます。公開Pages版へ実名、連絡先、証憑、会計情報を入力しないでください。

## 現在の導入状態

コード上の共有正本・会計・成果物機能と、実環境で安全に利用できる状態は同じではありません。確認済みのGoogle版は現在も `Authorization needed` で停止しており、所有者認可、実際のDrive権限、複数担当者、印刷・PDF、失敗時復元を含む結合試験は完了していません。認可と本書末尾の試験が終わるまでは、本番入力・請求発行・証明書交付に使用しません。

Apps ScriptのWebアプリはデプロイ所有者権限で実行します。現在のマニフェストは `ANYONE` ですが、これは通信上の入口であって操作権限ではありません。`doGet` と全APIの共通ゲートは、初期設定前は実利用者と所有者が一致しない要求を拒否し、設定後は `Session.getActiveUser()` のメール、有効ロール、必要権限を確認します。メールを取得できない場合にデプロイ所有者へ置き換えません。個人Googleアカウントが所有者の場合、コードは所有者本人以外を有効ロールへ追加することを拒否します。旧版から有効な別ドメインロールが残っていても、利用時に現在の所有者・同一Workspace制約を再検査して拒否し、正規管理者だけがそのロールを無効化できます。

複数担当者運用は、スクリプトと専用Drive資産の所有・管理を同一Google Workspaceへ移し、同一ドメインの利用者だけを登録し、Webアプリを `DOMAIN` 範囲で再デプロイして実アカウントの本人識別・許可・拒否を試験した後に開始します。現在の `ANYONE` デプロイを、そのまま安全な複数担当者運用として扱いません。

参照マニュアルにDIPS等の認証情報を平文で置かないでください。既に記載した認証情報は運用開始前に変更し、文書から削除し、共有履歴・閲覧者を確認します。本システムはDIPS認証情報を保存も使用もしません。

ロールは次の4種類です。全ロールに対象者共有正本の読取権限があるため、閲覧者にも個人情報を見せてよいかを確認してから登録します。正式会計の全状態を取得・操作できるのは `admin` と `accounting` だけで、`renewal` と `viewer` には返しません。更新担当が請求帳票を作る経路もなく、帳票作成時の正式請求参照はサーバー内部で検証します。

| ロール | 主な操作 |
|---|---|
| 管理者 `admin` | 初期設定、利用者、取込、バックアップ、対象者、会計、全成果物と成果物設定 |
| 更新担当 `renewal` | 対象者・講習項目の更新、請求帳票以外の成果物作成。経理項目の更新は不可 |
| 経理 `accounting` | 正式会計の更新、発行済み正式請求からの請求帳票作成。対象者・講習項目の更新は不可 |
| 閲覧 `viewer` | 閲覧のみ |

## 利用できる成果物作成

- 修了証明書発行台帳
- 更新講習修了証明書
- DIPS提出CSV
- 更新講習案内
- 講習記録簿
- 見積書・請求書

原本テンプレートは変更せず、出力先へコピーまたは新規ファイルを作成します。公開APIはブラウザ内の対象者本文を正本として受け取らず、`recordId`、画面で確認した版、内容ハッシュを受けて共有正本を再読込します。同じ正本・成果物種別・入力内容で再実行した場合は既存成果物を返し、入力が変わった場合は訂正版として版を進めます。請求帳票は対象者内の旧金額欄ではなく、正式会計で発行済みとなった請求IDを必須とします。請求帳票の同一性は対象請求の不変キー・発行時スナップショット・明細で判定し、別顧客の入金等による会計全体ハッシュの変化では新版を作りません。全額取消済みの請求から通常請求書を新規生成する操作は停止し、既存成果物の履歴参照だけを残します。

成果物の保存先は、承認済みのDriveフォルダ [2026年度](https://drive.google.com/drive/folders/1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN)（ID `1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN`）へコードと画面の両方で固定しています。対象は2026年度（2026年4月1日から2027年3月31日まで）で、対象者年度または入力済みの講習修了日が別年度なら誤保存を防ぐため停止します。初回の作成確定時にシステムが識別情報付きの `自動作成` フォルダを作り、その配下へ生成台帳、年次発行台帳、対象者別フォルダ、成果物を保存します。`自動作成` や対象者別フォルダを手作業で先に作成・移動・改名しないでください。別フォルダIDはサーバーで拒否します。旧出力先は直下が空、自動作成登録履歴なし、未解決の削除失敗なしを確認できた場合だけ固定先へ切り替えます。1件でも残存・履歴・確認不能があれば、自動移動・自動削除せず、監査付き移行が完了するまで停止します。2027年度へ切り替えるときは、承認済みの年度フォルダ、コード固定値、権限、結合試験を同じ変更として扱います。

成果物設定と振込先は別々に更新せず、版番号・変更hash・更新者・旧保存先履歴と一緒に1個のV2 Script Propertyへ確定し、直後に完全一致を読戻します。画面は読込時の設定版を送るため、別担当者の更新を上書きしません。同じ送信の再試行はidempotency keyで重複確定を防ぎます。変更hash・版・更新者は共有正本の監査行にも記録し、設定だけ確定して監査応答を失った場合は次回更新前に同じcorrelation IDで回収します。V2を完全検証した後だけ、旧V1設定と旧平文振込先を削除して全Script Propertiesを読戻します。削除結果を確認できない場合は確定済みV2を維持し、旧値を推測利用せず、次の設定更新と専用原本準備を停止します。V2が存在する場合は、破損していても旧V1設定へ推測で戻りません。なお、監査hashは共有正本バックアップへ入りますが、V2設定本文そのものは現行バックアップに含まれないため、設定の完全な災害復元は未完了です。管理者は変更後の設定画面と旧保存先履歴を確認し、別途安全な管理記録を保持してください。

Drive項目は、作成・コピー要求の時点で親フォルダを指定し、Workspaceの既定公開を無効にします。Drive要求を送る前に、操作内容をScript Propertyへ `ATTEMPT_STARTED` として保存・読戻しし、作成またはコピーの応答を失った場合は `OUTCOME_UNCERTAIN` のまま新規作成を停止します。応答を取得してID・親・種類・権限を検証できた場合も `CREATED_VERIFIED` を残し、成果物台帳、バックアップ台帳、設定ID、または災害復旧後の正本ポインタへそのIDを確定して完全一致を読戻した後だけ解除します。Drive一覧が一時的に0件でも未作成とは断定しません。対象者情報を書き込む前に、制限付き共有、親1か所、許可メールとの完全一致、domain/anyoneなし、編集者再共有なしを確認します。CSVは空ファイルのACL検査後に本文をアップロードします。専用原本フォルダと専用原本ファイルは所有者だけの権限で作成・再検査します。

成果物を作成しただけでは、修了証明書を「交付済」、見積書・請求帳票を「送付済」、DIPS修了者情報を「連携済」、本人手続を「更新申請済」にはしません。請求帳票は正式会計ですでに発行済みの請求から作り、帳票作成によって発行取引を増やしません。交付、送付、機関連携、本人申請の事実は、実際の業務を確認して別々に記録します。

## 初回導入の順序

順序を変えると、所有者誤認、旧データの重複取込、未承認採番、個人情報入り原本の利用につながります。

1. 初回セットアップ前に、複数担当者のWorkspace運用か、個人Googleアカウント1名だけの単独運用かを決める。共有正本は初期所有者を監査上固定するため、個人アカウントで先に初期化してからWorkspace所有者へ自動昇格・自動移管することはできない。
2. 複数担当者で使う場合は、認可・初期セットアップ・実データ入力より先に、Apps Scriptプロジェクトと固定保存先 `2026年度` を実運用する同一Google Workspaceの所有者管理下へ移す。固定フォルダIDを維持できない移管方法は使わず、新IDになる場合はコード固定値、権限、文書、結合試験を同じ変更として再承認する。個人所有の専用共有正本やFinanceStoreを先に作らない。Workspace所有者で `DOMAIN` 範囲へ再デプロイしてから次へ進む。
3. 最終的なデプロイ所有者がApps Scriptエディタの実行関数一覧から `ownerAuthorizeDeployment()` を実行し、表示されたGoogle同意画面を本人が確認する。再実行して `authorized: true` を確認した後、Workspace運用では `ownerSetupWorkspaceSystem()` を1回だけ実行する。これらの引数なし管理入口はエディタで選択できる必要があるため公開関数だが、Web画面からは一切呼ばず、内部の末尾 `_` 関数で実行者とデプロイ所有者の一致、固定保存先 `2026年度` の所有者一致を毎回検査する。画面の初期設定ボタンも常に無効である。初期化前に、Google発行IDトークンの発行元・本人メール・主体・発行時刻・有効期限が正しいこと、Workspaceモードでは `hd` クレームがメールドメインと一致することを検査する。IDトークンは保存・表示・ログ出力しない。個人Googleアカウントでは標準セットアップを拒否する。将来も1名だけで使うと決めた場合に限り、Apps Scriptエディタから `ownerSetupPersonalSingleUserSystem()` を明示実行する。この関数は内部で単独運用の固定確認値を渡すため、関数名を確認して選択する。元の3スプレッドシートとは別に、非公開フォルダ、共有正本、バックアップフォルダ、FinanceStore V2会計台帳を新規作成し、既存資料や旧FinanceStore V1を上書き・転用しない。認可URLやトークンをWeb画面、文書、チャットへ保存しない。セットアップ後、Workspace運用では同一ドメインのメールとロールを登録し、担当者ごとに実メール、許可操作、拒否操作を確認する。
4. 旧Apps Script版と同じオリジンにブラウザデータがある場合は先にJSON退避し、「ブラウザデータ移行を検査」で新規追加・同一内容を確認する。現行キーと旧版キーが同時にあり、正規化後の内容が一致しない場合は、移行・削除を停止して退避JSONを照合する。旧対象者データはinsert-onlyで、共有正本の既存行と異なる内容は停止する。旧案内日程もJSON退避・削除の対象に含めるが、対象者移行APIでは自動移行しない。退避内容を管理者が確認し、共有設定の4行へ手入力して保存・再読込する。既存対象者行を直す場合は、画面で確認したレコードID・版・内容ハッシュ付きの検証済みCSV取込を使う。検査後に旧データまたは共有正本が変わった場合は停止する。移行前バックアップを自動作成し、旧 `localStorage` は移行確定と同じ操作では削除しない。サーバー側の件数・ハッシュを別確認した後、同じApps Scriptオリジンで対象キーを削除し、不存在を再読込確認してから、完了済み移行バッチ・移行元ハッシュ・削除キー集合に結び付けた削除確認証跡をAPIで別記録する。

   旧データが公開Pages版のオリジンにある場合、Apps Script版からは読み取れない。公開Pages版の救出画面でJSON退避と同一オリジンの削除・不存在確認だけを行い、退避データは管理者が内容を確認して検証済みCSV等の承認済み経路で取り込む。公開Pages側の削除は共有正本の移行バッチへ自動記録されない。別ドメイン、別ポート、別ブラウザプロファイル、別端末の `localStorage` は読取・削除できない。
5. 更新予定者一覧を取り込む場合は年度を選び、「読取専用プレビュー」を確認してから確定する。対象は承認済みの `2026年名簿` と `2027年名簿` だけで、列構造・sheetId・ハッシュが一致しなければ停止する。確定時にも元シートを再読込し、元シートには一切書き戻さない。任意CSVは画面のテンプレートを使い、画面検査とサーバー全件検査の両方に合格したものだけを確定する。差分がある場合だけ取込前バックアップと新世代を作り、差分0件は無操作として返す。
6. 「専用原本を自動準備」を実行する。作成された証明書・発行台帳専用原本を、画像、注釈、コメント、ヘッダー、フッター、非表示領域を含む全ページで管理者が目視確認する。機械検査だけで個人情報ゼロは保証しない。確認前は運用原本として承認しない。
7. データ管理画面で、発行者情報、連絡先、登録番号、振込先、成果物アクセス許可メールを設定する。成果物の親フォルダは承認済みの [2026年度](https://drive.google.com/drive/folders/1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN) に固定され、利用者は変更できない。リンク・ドメイン公開を禁止し、許可メールを実際のDrive権限と完全一致させ、編集者による権限変更・再共有を無効にする。Googleグループ権限を使う場合も、各操作担当者には本人メールの直接 `user` 権限を付与し、許可一覧へ本人メールとグループメールの両方を記載する。旧保存先の警告が出た場合は設定保存やファイル移動を自己判断で進めない。旧フォルダが空で履歴・削除失敗がない場合は保存時の機械監査後に切替できるが、残存項目や履歴がある場合は担当部署が監査付き移行を行う。
8. DIPS担当部署へ5営業日の起算日と閉庁日を確認し、確認日・確認者・追加閉庁日を保存する。2028年以降は内閣府公式CSVを担当者が取得し、「公式祝日CSVを検査して登録」する。将来年の祝日を推測しない。
9. 既存の証明書番号とDIPS番号を台帳・DIPS側で照合し、「採番移行を検査（変更なし）」を行う。推奨開始値、`reportHash`、切替年月を別担当者が原本と照合した後だけ、自動採番切替チェックと開始値を保存する。dry-run自体は設定・採番・Driveを変更しない。
10. 更新講習案内を作る場合は案内日程を4行登録し、設定を保存する。
11. 所有者がApps Scriptエディタで `ownerInstallDailyBackupTrigger()` を実行し、午前2時頃の日次バックアップトリガーが1件だけ存在することを確認する。既存の同名トリガーは置換される。手動・日次とも1つのシステム実行IDで対象者・ロールのStoreバックアップとFinanceバックアップを対にし、`COMPLETE` だけを成功とする。失敗通知、保存先、実ファイルのハッシュ検証は別途試験する。
12. 管理者が対象者、正式会計、成果物、バックアップ、復旧の結合試験をテストデータで完了してから本番切替を承認する。

## 日常操作

1. Apps Script版を開き、画面上部が「専用共有正本へ接続」となっていること、自分のロールを確認する。接続失敗時はブラウザ保存へ切り替わらず読取・保存を停止する。
2. 対象者を登録・更新する。別担当者が先に更新した場合は版競合で停止するため、再読込して差分を確認してから入力し直す。無効化は物理削除せず、削除フラグと監査履歴を残す。
3. 請求は「正式会計」で下書きを保存し、内容確認後に発行確定する。通常画面は更新講習の課税10%専用である。8%・非課税・不課税・対象外は、根拠資料、適用理由、承認者、承認日を持つ例外処理を使い、通常画面で推測しない。発行済請求は直接編集せず、取消・訂正取引を使用する。入金は入金取引として登録し、請求への消込を別取引で記録する。
4. 返金、相殺、貸倒、請求取消・訂正、消込取消、会計締め、任意仕訳・仕訳取消は理由コード付きで申請し、申請者と異なる経理または管理者が48時間以内に承認する。請求訂正は、サーバーが算出した元請求の有効残額を全額取り消して残額0を確認してから、新請求を追記する。承認画面では元請求総額、実取消額、新請求額を分けて封印・表示し、ブラウザ指定の部分取消明細は受け付けない。確定済みの入金・返金・相殺・貸倒を誤った場合も元取引を直接変更せず、入金反対 `REVERSE_RECEIPT`、返金反対 `REVERSE_REFUND`、相殺・貸倒の決済反対 `REVERSE_SETTLEMENT` を二者承認で記録する。金額・対象者・元種別はサーバー正本から引き継ぎ、ブラウザで変更しない。台帳版が変わった承認依頼は無効となるため再申請する。会計締め前は「会計台帳を完全検査」を実行し、現在版の全イベント再生が正常であることを確認する。
5. 対象者詳細から成果物を選び、「作成前検査」を実行する。検査はDrive書込や採番予約を行わない。請求帳票では対象者に一致する発行済み正式請求を選ぶ。合格後に作成を確定し、サーバーが返した最新の正本版・ハッシュ、成果物ごとの結果とリンクを確認する。ブラウザから作成結果を対象者正本へ後書きしない。
6. 証明書交付、DIPSアップロード受理、本人申請、見積送付、請求帳票送付はファイル作成と別の事実として、実施日・確認者・使用版を記録する。請求発行自体は正式会計の発行操作として記録する。
7. 作業終了時と重要な取込・締め前に「共有正本をバックアップ」を実行し、1つのシステム実行に対するStoreバックアップとFinanceバックアップの組が `COMPLETE` であることを確認する。実行は冪等キーで識別し、同じキー・同じ内容の再送は同じ結果を返す。`PARTIAL` は片方まで作成された未完了、`ABORTED_PARTIAL` は途中状態を証拠として固定して新しい試行へ移った状態であり、どちらも成功扱いしない。闇雲に新しいキーで作り直さず、実行ID、StoreバックアップID、FinanceバックアップID、状態を照合してから再実行する。`COMPLETE` 後もブラウザの保留中冪等キーを消去して再読込確認できない場合は、新しいバックアップを作らず、同じ実行IDの一覧照合を先に行う。

作成処理は `LockService` で直列化し、Drive書込前に生成台帳へ `prepared` 予約を永続化します。中断時は予約とDriveの親、名前、識別マーカー、内容ハッシュ、Drive版を再照合し、完成済みが1件だけなら同じ予約を `created` へ確定します。前回実行の予約に一致する候補が0件でも、Drive一覧の反映遅延と作成直前中断を区別できないため新規作成せず停止します。確実な失敗は `error` として版を残し、結果不明・候補複数では重複作成せず担当部署確認で停止します。1種類が失敗しても、ほかの成果物の結果は個別に返します。見積・請求番号は既存CDP管理システムと衝突しない `QT-UC0157-...` / `INV-UC0157-...` を使います。

完全削除または台帳rollbackに失敗した場合は、対象の不透明なDrive ID、種別、台帳行・版、ファイル名等のハッシュ、エラーコード・エラー本文ハッシュだけを未解決記録へ保存し、その後の全成果物・専用原本の新規作成を停止します。氏名を含み得るファイル名・URL・生のACLエラー・メールアドレスはScript Propertiesへ複製しません。復旧担当者は記録を先に退避し、記録されたIDの実体、識別マーカー、親、共有権限と生成台帳行を照合します。誤作成物の完全削除または承認済み退避、台帳rollback、再読込による不存在・整合確認、復旧理由・実施者・確認者の記録がすべて終わった後だけ、該当する1個の `RENEWAL_ARTIFACT_CLEANUP_FAILURE_<ID>` を個別に解決します。接頭辞に一致するpropertyの一括削除、内容未確認でのproperty削除、同名ファイルを根拠にした推測復旧は禁止します。

共有権限の完全照合には、マニフェストで有効化した Advanced Drive service v3 を使用します。Drive APIが無効、権限一覧のページ取得が未完、`domain` / `anyone` 共有、許可外のuser/group、編集者による再共有がある場合は、個人情報を出力しません。許可したGoogleグループのメンバー管理は別途運用監査が必要です。

この版より前に作った `自動作成` フォルダ、対象者別フォルダ、生成台帳、年次発行台帳には固定識別マーカーがないため、自動で自己認定・再利用しません。内容と保存先を管理者が確認し、別名で退避して新規生成するか、承認済みの明示移行を行ってください。既存ファイルの説明欄や見出しを自動上書きして移行済み扱いにはしません。

## 停止中または手動確認が必要な連携

- 更新予定者一覧は明示的なpreview/commitだけを行い、定期自動取込・自動書戻しは行わない
- 講習記録簿・発行台帳など他の参照用スプレッドシートは自動取込・自動更新しない
- Googleドキュメントの6タブは承認済み版を参照元として固定し、マニュアル変更を自動反映しない
- メール送信
- DIPSへのログイン、CSVアップロード、申請操作
- 日次通知トリガー
- 会計災害復旧の自動実行・日常的な過去版への巻戻し。災害復旧は最新の登録済み `COMPLETE` バックアップに限り、管理者2名の明示操作でだけ行う

指定された3つのスプレッドシートは固定した項目対応の参照元です。更新予定者一覧は承認済みの2026年・2027年シートだけを明示preview/commitで読取専用取込し、講習記録簿と修了証明書発行台帳は成果物テンプレートの参照に使います。どの参照元にも書き戻しません。案内・講習記録簿も、作成時に想定外タブ、実データ、差込構造の変更を検査して停止します。原本編集者を限定し、変更は承認と試作出力の確認後に反映してください。

ただし、指定された修了証明書原本には実名タブ、発行台帳原本には既存受講者データがあるため、その2ファイル全体は生成原本として使用しません。「専用原本を自動準備」は、証明書を一時コピー上で単一ベース化・固定ダミー値化した後、清浄化済み状態を二次コピーして最終原本にします。一時コピーは完全削除を試み、最終原本の版履歴へ元の個人情報を持ち込みません。台帳はファイル全体をコピーせず、空の`ベース`シートだけを新規ブックへコピーします。両原本の機械検査を通るまで成果物作成を停止します。

DIPS機能は11列のUTF-8 BOM付きCSVを作るところまでです。列7は `PA000000000000`、状態フラグは新規・訂正・削除に対応し、訂正・削除では元の申請者IDを必須にします。CDPが行う「修了者情報連携日」と、受講者本人が行う「DIPS更新申請日」は別項目です。5営業日は前者の期限で、後者の通常期間は現免許期限の6か月前から1か月前までです。公開資料だけでは5営業日の起算日・閉庁日の扱いが確定しないため、翌営業日を1日目とする日付は暫定表示とし、必ず担当部署へ確認します。ログイン後に取得する最新の公式CSVひな形との最終照合も必要です。DIPSの認証情報は保存も使用もしません。

身体適性資料の期限は資料の記載日を入力し、CDPの確認日から自動計算しません。修了証明書の有効期限は初回発行日を基準とし、訂正・再発行日で初回発行日を変更して延長しません。現版は再発行日の別管理に未対応のため、初回正式成果物の作成後に発行日・期限を変更する操作は停止します。

正式会計は、請求・明細、入金、消込、取消・決済、締め期間、仕訳を別データとして保存します。FinanceStore V2の正本はハッシュ連鎖した追記専用 `finance_events` です。現在状態は1セル上限を避ける30,000文字単位のチェックポイント、一覧表は再生成可能な投影です。イベントを先に確定し、その後のチェックポイント・投影・監査の失敗は `committed: true` と `recoveryNeeded` で通知し、次回読込で決定論的に修復します。会計イベント、承認申請、承認実行は、同じ利用者操作の再送に同じ冪等キーを使います。同じキー・同じ内容は元の結果を返し、同じキー・異なる内容は拒否します。送信結果が不明な場合は、新しいキーで作り直さず、画面の安全再送または一覧再読込で照合します。

正式請求の発行時は、ブラウザから発行者・登録番号・振込先を受け入れません。サーバーが共有正本の請求先と非公開の事業者設定を読み、宛先、敬称、住所、発行者、連絡先、適格請求書発行事業者登録番号、振込先を請求イベントへ封印します。発行後に対象者や事業者設定を変更しても、発行済み請求書・対の見積書はこのスナップショットだけから再現します。

通常読込ではイベント連鎖、現在状態、チェックポイント、投影を検査し、欠落・不整合があれば投影等を再構築します。全履歴再生は24時間ごとの健全性検査または明示検査で行います。会計バックアップは全イベント、チェックポイント、監査、承認履歴を含む独立した復旧可能なスナップショットで、作成時はレジストリを `PREPARED` から `COMPLETE` へ確定し、本文ハッシュと完全再生を検証します。対象者バックアップに同梱される会計情報は相互照合用であり、対象者・ロール復元から会計を上書きしません。

税区分は課税10%・課税8%・非課税・不課税・対象外を分け、会社固定の端数方式で請求書内の税率別合計に1回だけ端数処理します。10%以外は明細ごとに根拠資料、適用理由、承認者、承認日がなければ会計エンジンが拒否します。下書きは債権計上せず、発行済み請求は直接上書きしません。

現在の画面は、請求下書き・発行、1入金の複数請求への消込、未消込入金の後日消込、消込取消、請求全額取消・訂正版同時発行、返金・相殺・貸倒、締め申請、承認、全イベント再生検査を提供します。任意仕訳・仕訳取消など、会計エンジンが保持できるすべての操作を画面から直接行えるわけではありません。CSV出力は補助であり、税務申告・法定帳簿・会計ソフト連携を代替しません。売上計上時期、勘定科目、税込経理・税抜経理、貸倒要件は税理士と承認した会社方針を正とします。

## データの位置づけ

Apps Script版の対象者・講習項目は専用共有正本へ保存し、対象者レコードは `管理ID | 年度 | 受講回数` を業務キーとして版管理します。対象者と会計は別の正本です。対象者とロールを全件置換する取込・復元は、2枚の新しい世代シートへ書込・検証してから `_meta` の `activeDataGeneration` を1回だけ切り替えるため、片方だけ新世代になる状態を正本として読みません。正本との差分が0件なら、バックアップ、世代シート作成、ポインタ切替を行わない無操作として返します。切替前の世代は削除せず保持します。

ロールバック用の旧世代は16件から管理画面に警告し、24件に達した時点で新しい一括取込・対象者復元だけを安全停止します。中断で残った未完了世代も保持数へ含めます。さらに、[Google Sheetsの公式上限は1ファイル1,000万セル](https://support.google.com/drive/answer/37603)のため、全シートの割当セルを実測し、700万セルから警告、作成予定の2シート、世代台帳、初回世代ポインタ、バックアップ台帳、未完了バックアップ再調整、変更件数分のPREPARED／COMMITTED監査行を加えて800万セルを超える場合は、バックアップ・監査・バッチ状態を変更する前に停止します。各台帳と新世代の予定行数が20万行を超える場合も同じ位置で停止します。通常の1件登録・更新・閲覧は継続できます。シートを手作業または自動で削除せず、管理者が正本、保留中処理、復元候補、バックアップを照合した承認済みの保管・圧縮手順を別途実施します。現版は旧世代の自動削除・自動圧縮を行いません。

正式会計設定後、請求額、税率、請求番号、請求状態、入金情報などの対象者内ミラーは表示・成果物連携用に限ります。ブラウザ保存や取込で送られたミラー値は破棄し、FinanceStoreの発行済み請求ID、対象者ID、不変キー、明細、税、端数方式と一致するサーバー内部反映だけを許可します。FinanceStoreが唯一の変更可能な会計正本であり、対象者レコードを第二の会計正本にしません。

公開Pages版の通常表示は、合成サンプルをメモリ上だけで扱う閲覧専用です。対象者・監査・会計を `localStorage` へ保存せず、JSON復元やサンプル置換も無効です。旧保存は明示した退避・削除操作だけが同一オリジンで扱います。Apps Script版でサーバー接続に失敗した場合も、`localStorage` へ黙って保存せず停止します。ブラウザCSV・JSON、Drive成果物、生成台帳は対象者正本・会計正本を代替しません。

共有正本は初期作成直後から、対象者・ロールを件数・行ハッシュ・全体ハッシュ付きのactive generationへ切り替え、通常の1行更新でもPREPARED回復証明と世代ハッシュを同時更新します。再試行時はPREPARED時の作成日時・更新日時と日時以外の全セルハッシュを再利用し、内容・行・世代のどれかが違えば停止します。対象者・ロール用バックアップは、登録済みDriveファイルと内容ハッシュに加え、ファイル自体が非公開で直接の編集者・閲覧者・コメント権限を持たないことを、作成直後、冪等再利用、読込、復元時に全権限ページで検査します。Apps Script版の管理者画面から、申請管理者が理由と別の管理者を指定して予行し、その指定を受けた管理者本人だけが15分以内に、保留一覧の正確な `batchId` と明示確認を再入力して確定または却下できます。確定直前に復元前安全バックアップを作り、対象者・ロールを検証済みの新世代へまとめて切り替えます。旧世代は保持し、準備途中の世代は正本として読みません。バックアップ内の有効ロールにも、復元時点の同一Workspace・個人所有者制約を再適用します。公開Pages版では復元できません。実Driveでの強制終了・復元試験は未実施です。

対象者・ロールの単票更新は、監査の `PREPARED` を先に記録し、正本行を読戻してから `COMMITTED` を確定します。正本行の保存後に監査確定だけが失敗した場合は、保存済みを未保存とは表示せず `committed: true`、`recoveryNeeded: true`、監査確認IDを返します。画面は同じ内容を再登録せず、そのIDで監査だけを再照合します。次の更新前にも保留マーカーを検査し、正本と一致する場合だけ監査を自動確定します。別内容の未解決PREPAREDが同じ対象に残る場合は上書きを停止します。通常操作で利用者が入力した「承認者」は監査へ採用せず、実際の別管理者承認フローだけが承認者を記録します。

会計災害復旧は、登録一覧の最新 `COMPLETE` Financeバックアップだけを候補とし、バックアップ後の会計イベントがある場合と、正常な現台帳と内容が異なる過去版への巻戻しを拒否します。申請管理者が別の有効な管理者を指定し、その本人が15分以内に正確な依頼IDと確認文を再入力した場合だけ進みます。元の専用Storeを直接上書きせず、専用データフォルダ内にStore全体の非公開ステージコピーを作り、会計イベント・チェックポイント・監査・承認・投影、対象者参照、請求成果物参照、両管理者を検証します。復旧後ベースラインバックアップをステージ側で作成・検証してからScript PropertyのStoreポインタを1回切り替え、旧Store IDを安全退避先として削除せず保持します。応答喪失時は保留・完了一覧と有効Store IDを照合し、盲目的に再実行しません。この処理はコード実装済みですが、実Driveでの障害復旧試験は未実施です。

旧FinanceStore V1の `finance_snapshots` はV2として開かず停止します。V1からV2への移行は、旧台帳を凍結し、別の新規V2台帳へ二者承認で移し、全件照合する独立した設計・承認・試験が必要です。

- 正式会計の分割入金、未消込、過入金、複数請求への消込、返金、相殺、貸倒、締め、仕訳、訂正を経理・税理士の期待値と照合する
- 入金・返金・相殺・貸倒それぞれの反対取引、自己承認拒否、別担当者承認、再送冪等性を経理・税理士の期待値と照合する

これらが完了するまで、コード上の正本機能を実データの本番正本または正式会計帳簿として扱わないでください。

## 実データ連携を広げる場合

実データ連携を追加する場合は、接続先、所有者、読取範囲、書込範囲、表示項目、メール送信、権限、バックアップ、デプロイ可否を事前に確認し、別途レビューします。現在の書込先は、明示初期化した専用共有正本・専用バックアップ、コードで固定した承認済み成果物保存先と生成台帳、専用原本です。固定された `2026年度` は成果物系だけの親であり、共有正本、FinanceStore、バックアップ、専用原本はそれぞれ別の管理領域です。元の3スプレッドシート、Googleドキュメントのマニュアル、DIPSには書き込みません。
