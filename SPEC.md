# CDP免許更新管理 仕様

## 1. 目的

無人航空機操縦者技能証明の更新漏れを防ぎ、更新期限、講習、修了証明書、講習機関のDIPS修了者情報連携、本人のDIPS更新申請、見積・請求を混同せずに管理する。保存済み対象者レコードから、更新講習業務に必要な6種類の成果物を、既存のひな形に沿って作成できるようにする。

## 2. 現在の運用範囲

### 対象者データ

- Apps Script版の対象者データは、初回セットアップで新規作成した非公開の専用スプレッドシートを共有正本として保存する。既存の参照スプレッドシートを正本へ転用しない。
- 公開GitHub Pages版は、合成サンプルをメモリ上だけで表示する閲覧専用デモとする。通常起動・一覧・検索は旧 `localStorage` を読み込まず、対象者・監査・会計データを永続化しない。旧版で同じオリジンに残ったデータは、利用者が救出ボタンを明示した場合だけJSON退避・削除のために読み取る。実名、連絡先、証憑、会計情報を新規入力しない。
- Apps Script版で共有正本の接続、Google認可、利用権限のいずれかを確認できない場合は読取・保存を停止し、`localStorage` へ自動フォールバックしない。
- レコードの業務キーは `管理ID | 年度 | 受講回数` とする。同一人物の別年度・別受講回は別レコードとし、氏名だけで同一人物判定しない。
- 新規保存は期待版0、更新は画面が読んだ版番号と共有正本の現版を一致させる。競合時は上書きせず再読込を求める。
- 削除は物理削除せず無効化し、版と監査ログを進める。旧請求欄または正式会計に請求・入金等の痕跡があるレコードは、管理者を含め無効化を拒否し、業務状態で対象外等を管理する。
- 更新予定者一覧は承認済みの `2026年名簿` / `2027年名簿` だけを、管理者がpreview/commitで明示取込できる。列名、sheetId、元ファイル更新時刻、行ハッシュを検査し、確定時にも再読込する。元シートへは書き込まない。
- 旧Apps Script版と同じオリジンの `localStorage` は、管理者が全件preview/commitで共有正本へコピーする。現行キーと旧版キーが併存し、正規化後の内容が一致しない場合は、どちらかを推測で採用せず移行・削除を停止する。移行は新規行の追加と同一内容のスキップだけを許し、既存行と内容が異なる場合は停止する。既存行の変更は、画面で確認した `recordId`・版・内容ハッシュを行ごとに付けた検証済みCSV取込を使用する。検査後に移行元または共有正本が変わった場合は停止し、確定前に共有正本の安全バックアップを作る。
- 旧データは移行確定と同時に自動削除しない。旧案内日程もJSON退避・削除のキー集合へ含めるが、対象者移行APIから自動移行せず、管理者が内容を確認して共有設定へ手入力・再読込する。JSON退避と正本の件数・ハッシュを別確認した後、同じApps Scriptオリジンで対象キーを削除して不存在を確認し、完了済み移行バッチ、移行元ハッシュ、キー集合版、削除件数に結び付けた削除確認証跡をAPIで別記録する。サーバーはブラウザ保存の実削除を直接検査できないため、この証跡は管理者の確認記録である。
- 旧公開Pages版の `localStorage` はApps Script版から読めない。公開Pagesの救出画面で同一オリジンのJSON退避・削除・不存在確認を行い、退避データを検証済みCSV等の承認済み経路で取り込む。公開Pages側の削除は共有正本の移行バッチへ自動記録しない。別オリジン・別ブラウザプロファイル・別端末の保存は操作できない。
- メール送信、日次通知、DIPS操作は行わない。日次バックアップは、所有者がApps Scriptエディタで `ownerInstallDailyBackupTrigger()` を明示実行した場合だけ午前2時頃に動作する。
- `localStorage`、ブラウザCSV・JSONは共有正本や正式会計帳簿ではない。

### Google認可と利用者

- Webアプリは `USER_DEPLOYING` でデプロイ所有者として実行する。所有者はApps Scriptエディタの実行関数一覧から `ownerAuthorizeDeployment()` を実行し、必要スコープの同意を本人が完了する。エディタ表示用の引数なし管理入口は公開関数だがWeb画面からは呼ばず、内部の末尾 `_` 関数で実行者とデプロイ所有者、固定保存先所有者の一致を毎回検査する。実デプロイ設定が異なる場合にも任意利用者を初期管理者にしない。
- 現在のマニフェストは `ANYONE` だが、これを操作権限としない。`doGet` と全APIの共通ゲートは、初期設定前は実利用者と所有者の一致、設定後は `Session.getActiveUser()` のメール、有効ロール、必要権限を検査する。メールを取得できない場合に `getEffectiveUser()` の所有者へ置き換えず、安全側で停止する。
- 個人Googleアカウントが所有者の場合、所有者本人以外を有効ロールへ追加する処理は拒否する。複数担当者運用は、初回認可・初回セットアップより先に、スクリプトと固定保存先を同一Google Workspaceの所有者管理下へ移し、同一ドメインの利用者だけを登録し、Webアプリを `DOMAIN` 範囲で再デプロイして実IDの取得・許可・拒否を確認した後に限る。共有正本は初期 `createdBy` を監査上固定するため、個人アカウントで作成した正本をプロジェクト移管だけでWorkspace正本へ自動昇格しない。既に個人所有で初期化済みなら、別のWorkspace正本への監査付き移行を独立して設計・承認する。旧版から残った有効ロールも利用時に現在の所有者・同一ドメイン制約を再検査し、不適合な本人は拒否する。正規管理者はその行を無効化できる。共通ゲートがあっても、現在の `ANYONE` デプロイを安全な複数担当者運用とみなさない。
- ロールは `admin`、`renewal`、`accounting`、`viewer` とする。更新担当は対象者・講習、経理は会計、管理者は初期設定・権限・取込・バックアップを担当する。正式会計の全状態を取得・操作できるのは `admin` と `accounting` だけとし、`renewal` と `viewer` へFinanceStore状態を返さない。
- 全ロールに共有正本の読取権限がある。個人情報の列単位・項目単位マスキングは未実装のため、閲覧ロールも業務上必要な担当者だけに付与する。
- 初回共有正本セットアップは、ログイン利用者とデプロイ所有者が一致する場合だけ実行し、初期管理者として登録する。有効な管理者を0名にはできない。Workspace複数担当者モードは個人メールドメインを拒否し、初回Drive書込より前に停止する。
- 最終所有者はApps Scriptエディタで `ownerAuthorizeDeployment()` を実行して `authorized: true` を確認する。Workspace運用では、その後 `ownerSetupWorkspaceSystem()` を1回だけ実行する。初期化前に、固定保存先の所有者、Google発行IDトークンの本人メール・検証済みフラグ・発行者・主体・発行時刻・有効期限、Workspaceの `hd` クレームとメールドメインを一致検査する。トークンは保存・表示・ログ出力しない。個人Googleアカウントの単独運用は将来のWorkspace自動移管がないことを理解し、Apps Scriptエディタで `ownerSetupPersonalSingleUserSystem()` を明示選択した場合だけ初期化する。日次バックアップを開始するときだけ `ownerInstallDailyBackupTrigger()` を実行する。これらは通常利用者用Web画面から一切呼ばない。

### 成果物データ

- 保存済みレコードに対するユーザーの明示操作だけで Drive に出力する。公開APIは `recordId`、画面で確認した `expectedVersion`、`expectedPayloadHash` を必須とし、ブラウザのレコード本文を正本にしない。サーバーは作成前とロック取得後に共有正本を再読込し、版またはハッシュが変わっていれば停止する。
- 作成前検査と作成確定を分ける。検査では書込や採番予約を行わない。
- テンプレート原本は読み取り専用とし、必ずコピーへ記入する。
- 修了証明書と発行台帳は、他受講者情報を含まない専用原本IDを必須とする。実名タブまたは既存データを含む参照元の直接コピーを運用原本にしてはならない。証明書の自動準備は、一時コピーを単一ベース化・固定ダミー値化し、その清浄化済み状態の二次コピーだけを最終原本として採用する。一時コピーの削除に失敗した場合は停止してIDを報告する。台帳は参照元ファイル全体ではなく、空の`ベース`シートだけを新規ブックへコピーする。
- `LockService` で同時作成、採番、生成台帳更新を直列化する。
- 対象レコードID、成果物種別、入力ハッシュで冪等性を判定する。同一内容は既存成果物を返し、変更内容は新しい版として作る。作成確定時に必要となった番号・計算日・正式請求参照は、Drive作成前に共有正本へサーバー側で予約保存し、ブラウザから作成後の書戻しを行わない。
- Drive作成前に生成台帳へ `prepared` の永続予約を追記し、対象、種別、入力ハッシュ、版、予定ファイル名、保存先、採番情報を固定する。個々のDrive作成・コピー要求の直前にも `ATTEMPT_STARTED` を永続化して読戻す。応答不明時は `OUTCOME_UNCERTAIN`、ID・親・種類・権限検証後から上位台帳・設定・正本ポインタの確定前までは `CREATED_VERIFIED` を保持する。耐久台帳またはポインタへ同一IDを保存して完全一致を読戻した後だけ解除する。応答喪失・中断後は予約とDriveの親、名前、識別マーカー、内容ハッシュ、Drive版を再照合し、一致が1件だけなら `created` へ確定する。前回実行の予約がある場合は候補0件でも一覧反映遅延と未作成を区別できないため再作成しない。結果を一意に判断できない場合は重複作成せず「担当部署に確認が必要」で停止する。確実に失敗した予約は同じ行を `error` へ確定して版を消費し、版番号を黙って再利用しない。完全削除・台帳rollbackを確認できない失敗は、対象ID・台帳位置と、氏名等を含まないハッシュ化済み診断情報だけを未解決記録へ残し、個別復旧が完了するまで全成果物・専用原本の新規作成を停止する。
- 成果物ごとに成功・既存利用・失敗を返し、一部失敗を全件成功として扱わない。
- 個人情報の出力先は承認済みのDriveフォルダ `2026年度`（ID `1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN`）へ固定し、リンク公開・ドメイン公開のフォルダには作成しない。対象は2026年度（2026-04-01から2027-03-31）とし、対象者年度または入力済みの講習修了日の年度が異なる場合は停止する。初回の作成確定時に識別情報付き `自動作成` を作り、その配下だけへ生成台帳、年次台帳、対象者別成果物を保存する。別ID、識別情報のない手作業の同名フォルダ、監査未完了の旧保存先設定は拒否する。
- 固定化前の旧出力先、旧成果物、`prepared`・`error` を含む履歴は自動移動・自動削除しない。監査付き移行が完了するまで固定親への新規作成も停止する。
- 旧保存先からの切替時は、旧フォルダを取得でき、直下が空で、自動作成フォルダの登録履歴と未解決の削除失敗記録がないことを機械確認する。いずれかを確認できない場合は停止する。すべて空であると確認できた場合だけ設定を固定先へ切り替え、ファイルの移動・削除は行わない。監査済みの旧フォルダID・確認者・確認時刻・切替版はV2設定履歴へ残し、上書きで失わない。
- 新規フォルダ、Googleドキュメント／スプレッドシートのコピー、CSVは、Drive API要求で親を同時指定し、Workspaceの既定公開を無効にする。個人情報を書き込む前に、制限付き共有、親1か所、許可メールとの完全一致、domain/anyoneなし、編集者再共有なしを検査する。CSVは空ファイルを検査してから本文をアップロードする。専用原本は所有者だけのACLを検査する。
- 案内日程4行、採番移行設定、発行者設定、振込先は、版・更新者・変更hash・旧保存先履歴を含む単一V2 Script Propertyへ保存し、端末間で共通利用する。期待版とidempotency keyで競合・重複を防止し、書込後の完全一致とhashを読戻す。確定した変更hash・版・更新者は共有正本のCOMMITTED監査行へ記録し、監査未回収中は次版を停止する。V2破損時は旧設定へfallbackしない。監査hashはバックアップ対象だが設定本文の災害バックアップは別途未実装である。
- 証明書・DIPSの既存番号照合が未確定なら、空欄からの自動採番を停止する。
- 設定した自動採番切替年月より前の講習は、照合済み番号の手入力を必須にする。
- 見積書・請求書成果物は、対象者 `recordId` に一致する発行済み正式請求の `financeInvoiceId`、不変キー、発行時スナップショット、正式明細、有効請求残額が0より大きいことを必須とする。対象者レコード内の旧金額・請求状態を帳票金額の正本にしない。会計全体状態ハッシュは監査メタデータに限り、成果物版の同一性判定へ含めない。
- Googleドキュメントの6タブと6種類の成果物の対応は承認済み版を参照し、マニュアル本文を実行時に解析して自動追従しない。参照元の版・更新時刻が変わった場合は再承認まで停止する。
- 専用原本の機械検査は既知の差込欄・構造・権限を対象とする。画像、注釈、コメント、ヘッダー、フッター、非表示領域を含む個人情報不存在は全ページ目視で確認する。

## 3. 期限管理ルール

| 項目 | 基準 | システムで見ること |
|---|---|---|
| 現在の資格有効期限 | DIPSまたは技能証明書の有効期限 | すべての基準日 |
| 社内案内開始 | CDPが承認した運用日 | 法定期限と混同せず、設定値として管理する |
| DIPS更新開始 | 現有効期限の6か月前 | 通常の更新手続が可能になる日 |
| 申請期限 | 現有効期限の1か月前 | ここまでに更新申請を進める |
| DIPS修了者情報連携 | 修了証明書発行後5営業日以内 | CDP受講分の機関連携。起算日・閉庁日は担当部署確認が必要で、翌営業日を1日目とする値は暫定 |
| 講習修了証明書の期限 | 修了証明書初回発行日の3暦月後の前日 | 初回発行日基準で作成時に再計算する。初回発行日と実交付日は別に記録する |
| 更新完了日 | DIPSまたは新しい技能証明書で確認した日 | 講習日や成果物作成日とは別 |
| 新しい資格有効期限 | 更新後のDIPSまたは証明書に表示される期限 | 次回更新管理の正式な基準 |

```text
講習修了証明書の期限 = 修了証明書初回発行日 + 3暦月 - 1日
```

修了証明書期限は、3暦月後に同じ日があればその前日、同じ日がなければ対象月の末日とする。例: 1月31日発行は4月30日まで。業務日付は `YYYY-MM-DD` として扱い、交付後は実際の記載日付と実交付日を確認記録として残す。

## 4. アラートルール

更新確認期限は、その案件に入力されている有効な期限のうち最も早い日を採用する。

```text
更新確認期限 = min(現在の資格有効期限 - 1か月, 講習修了証明書の期限, 身体適性資料の記載期限)
```

身体適性資料期限は資料に記載された期限を転記し、CDPの確認日から自動計算しない。システム計算値は案内・検査用であり、DIPS、技能証明書、交付済み証明書の確認値を無断で上書きしない。

## 5. 自動作成する成果物

| 種別 | 出力 | 主な検査・処理 |
|---|---|---|
| 発行台帳 | Googleスプレッドシート | 証明書番号、対象者、区分、講習日、発行日、実交付情報、有効期限、備考を台帳様式へ記入 |
| 講習修了証明書 | Googleドキュメント | 対象者、技能証明申請者番号、区分、講習日、有効期限、担当講師、発行者を記入 |
| DIPS提出CSV | UTF-8 BOM付きCSV | 11列を固定順で作成。列7固定値、訂正・削除フラグ、公式ひな形との未照合警告を検査 |
| 更新講習案内 | Googleドキュメント | 対象者、免許期限、受講可能期間、資格区分、料金、4行の日程、連絡先を記入 |
| 講習記録簿 | Googleスプレッドシート | 回転翼航空機（マルチローター）に限定し、一等・二等の様式、日付、時間、担当者、会場、最低時間を検査 |
| 見積書・請求書 | Googleスプレッドシート | 発行済み正式請求の明細・取引年月日・税抜・税率別消費税・税込をV3レイアウトへ正確に転記 |

詳細なテンプレートID、項目、採番、出力先は `ARTIFACT_AUTOMATION.md` を正とする。

## 6. 作成前検査

共通で対象レコードIDと対象者名を必須とする。成果物別に次を検査する。

- 修了証明書、発行台帳、DIPS CSV、講習記録簿: 講習修了日、資格区分、受講開始時の本人・対象講習照合結果・照合日・照合者・証憑参照
- 修了証明書、発行台帳、DIPS CSV: 修了証明書初回発行日。証明書番号年月と期限は初回発行日基準
- 修了証明書: 技能証明申請者番号、担当講師、発行者情報
- DIPS CSV: CDP受講分、技能証明申請者番号、番号形式、11列の入力値、DIPS営業日暦の確認日・確認者・追加閉庁日
- 更新講習案内: 現免許期限、受講可能期間、料金、連絡先、4行の日程と時間帯
- 講習記録簿: 航空機の種類が回転翼航空機（マルチローター）、学科会場と各学科項目の日付・開始・終了・担当者。停止処分者向け講習が「あり」の場合だけ実地会場・実地項目・必要時間（一等演習5分、二等演習6分）を必須化
- 見積書・請求書: 対象者に一致する発行済み正式請求ID、不変キー、請求先、取引年月日（役務提供日）、正式明細、税区分、端数方式、合計、発行者、振込先。通常画面は課税10%だけを扱い、10%以外は明細ごとの根拠資料・適用理由・承認者・承認日がある例外処理に限る
- 共通設定: 固定保存先 `2026年度` の非公開・ID一致、承認済みアクセス許可メール、自動準備または管理者承認済みの無個人情報証明書・台帳専用原本、必要テンプレートの清浄性と読取権限、空欄から採番する場合の移行確認

更新講習案内の日程は4行すべてに日付・会場・1つ以上の受付時間帯を必要とし、対象者の講習実施可能日から講習締切日の範囲外はエラーにする。

新規の見積・請求番号は更新講習専用の `QT-UC0157-yyyyMMdd-N` / `INV-UC0157-yyyyMMdd-N` とし、既存CDP管理システムの番号空間と分離する。既存CDP側ですでに正式採番済みの番号を手入力する場合は、従来の `QT-yyyyMMdd-N` / `INV-yyyyMMdd-N` も受け入れる。

## 7. DIPS CSVの境界

DIPSに渡す11列CSVをCDP受講分について作るだけとし、ログイン、画面操作、アップロード、申請は行わない。列7は `PA000000000000` 固定、状態フラグは新規=`1`、訂正=`2`、削除=`3` とする。訂正・削除は誤登録時と同じ申請者IDを必須とし、自動採番しない。DIPSの認証情報をコード、設定、対象者レコード、生成台帳へ保存しない。

CSV作成日、講習機関が実際にアップロード受理を確認した「DIPS修了者情報連携日」、受講者本人の「DIPS更新申請日」を分離する。5営業日は機関連携の期限であり、本人申請の6か月前〜1か月前とは別である。公開資料だけでは営業日の起算と閉庁日を確定できないため、内閣府祝日・追加閉庁日・確認日・確認者が揃わない場合は期限を断定せず、DIPS CSV作成を停止して「担当部署に確認が必要」と表示する。

DIPSから取得する最新の公式CSVひな形とは未照合である。作成前検査と作成結果に警告を表示し、担当者が公式ひな形の列名・列順・コード値と照合してから手動で使用する。

祝日マスタはコード固定の2026年・2027年と、管理者が内閣府公式CSVを取得・確認して登録した年だけを使用する。2028年以降を曜日規則や過前年から推測しない。公式URL、CSVハッシュ、確認日、確認者を保存し、登録済み年の差替えには訂正理由を必要とする。祝日マスタとは別に、DIPS担当部署の追加閉庁日と5営業日の起算方法を確認する。

## 8. 見積・請求計算

対象者画面の旧互換欄と、標準画面が作る単一料金・値引の正式請求下書きは次の順に計算する。値引は税抜の絶対額であり、負数として入力しない。

```text
税抜対象額 = max(0, 料金税抜 - 値引税抜)
消費税     = 端数処理(税抜対象額 × 税率)
税込合計   = 税抜対象額 + 消費税
```

正式会計の複数明細では税率別税抜合計を作り、1つの適格請求書につき税率ごとに消費税の端数処理を1回だけ行う。端数方式は設定された切捨て・四捨五入・切上げを使い、明細ごとに税を丸めて合計しない。

請求書には請求日と別に取引年月日（役務提供日）を表示する。更新講習等は通常10%とし、標準画面は課税10%の請求だけを作成する。8%・非課税・不課税・対象外は明細ごとに根拠資料、適用理由、承認者、承認日が揃わない限り会計エンジンが拒否する。

請求帳票は、正式会計ですでに発行確定した請求からだけ作る。帳票ファイル作成は請求発行取引を新規作成せず、見積送付、帳票送付、入金消込も自動確定しない。ほかの成果物もファイル作成と交付・連携・申請を分けて記録する。

正式会計台帳は、対象者レコード内の旧請求・入金欄とは別に、`invoices`、`invoice_lines`、`payments`、`payment_allocations`、`credit_notes`、`closing_periods`、`journal_entries` を保持する。FinanceStore V2の唯一の正本は、コマンド、前後状態ハッシュ、前イベントハッシュ、操作者、承認参照、冪等キーを持つ追記専用 `finance_events` とする。現在状態は1セルの50,000文字制限を避ける30,000文字単位の `finance_state_chunks`、一覧シートは正本から再生成できる投影である。

会計コミットはイベントを最初に追記する。イベント確定後にチェックポイント、メタ情報、投影、監査の更新が失敗した場合は、未保存とは表示せず `committed: true`、`recoveryNeeded: true`、復旧コードを返す。利用者は同じ取引を手入力し直さず再読込し、イベント再生による派生データ修復を確認する。状態不明コードが出た場合は再送せずイベント正本を確認する。会計イベント確定、承認申請、承認実行は、1つの利用者操作と通信再送に同じ冪等キーを使い、同じキー・同じ内容は最初の結果を返し、同じキー・異なる内容は拒否する。送信結果が不明な場合は新しいキーで別申請を作らず、同じキーの安全再送または一覧再読込で照合する。

正式請求の発行時は、クライアントが送った `billingSnapshot` を正本にしない。サーバーが共有正本の請求先と非公開事業者設定から、宛先、敬称、住所、発行者、発行者住所・電話・FAX・メール、適格請求書発行事業者登録番号、振込先を正規化・検証して発行イベントへ封印する。発行済み請求の不変キーはこのスナップショットを含み、帳票と対の見積書は後日の対象者・会社設定を参照せず、封印値だけを使う。

FinanceStore設定後、対象者レコードに残る請求額、税率、請求番号、請求状態、入金情報等は表示・成果物連携用のミラーである。通常保存・旧ブラウザ移行・CSV取込で送られたミラー値は破棄し、既存の正本値を維持する。ミラーを変更できるのは、サーバーがFinanceStoreの発行済み請求ID、対象者ID、不変キー、明細、税率別計算、端数方式と再照合した内部反映だけである。FinanceStoreを変更可能な唯一の会計正本とし、対象者レコードとの二重更新を許さない。

通常読込はイベント連鎖、最新チェックポイント、投影、監査を検査・修復する。全イベント再生は24時間ごとの健全性検査または強制検査で実施する。会計バックアップは全イベント、チェックポイント、監査、承認履歴を含む独立JSONで、レジストリを `PREPARED` から `COMPLETE` へ確定し、本文ハッシュ、全イベント再生、状態ハッシュを検証する。

会社設定の消費税端数方式は切捨て・四捨五入・切上げのいずれかに固定し、取引開始後に変更しない。税区分は課税10%、課税8%、非課税、不課税、対象外を分ける。発行時に税率別合計で税額を計算し、請求・売掛金・売上・仮受消費税の仕訳を保存する。

請求は下書き、発行済みを分け、下書きは債権・会計集計へ含めない。発行済み請求、入金、消込、取消・決済、締め、仕訳は不変キーで検査し、直接上書きせず反対取引または訂正取引で処理する。請求訂正は元請求の有効残額をサーバーで算出して全額取り消し、残額0を検査してから新請求を追記する。ブラウザ指定の取消明細は拒否し、承認用封印では元請求総額、実取消額、新請求額を分ける。入金と消込を分けるため、分割入金、未消込、過入金、1入金の複数請求への配賦を表現できる。確定済みの入金・返金・相殺・貸倒を訂正するときは、元取引の金額や状態を上書きせず、入金は `REVERSE_RECEIPT`、返金は `REVERSE_REFUND`、相殺・貸倒は `REVERSE_SETTLEMENT` を作る。金額、対象者、元種別はサーバー正本から継承し、二重反転や元取引不整合を拒否する。

返金、相殺、貸倒、請求取消・訂正、入金・返金・相殺・貸倒の反対取引、消込取消、会計期間締め、任意仕訳・仕訳取消は二者承認対象とする。申請者本人は承認できず、承認期限は48時間、承認対象の会計版が変わった場合は `STALE` として再申請する。理由コード、申請者、承認者、前後版、ハッシュを監査へ残す。

現在の画面は請求下書き・発行、1入金の複数請求への消込、未消込入金の後日消込、消込取消、請求全額取消・訂正版同時発行、返金・相殺・貸倒、会計締め申請、二者承認、全イベント再生検査を提供する。締め申請は現在版の完全検査が済んでいなければ停止する。任意仕訳・仕訳取消など、会計エンジンが保持できるすべての操作を画面から直接行えるわけではない。管理年度別の集計画面は更新案件の参考集計であり、法定帳簿、会計ソフト、会計期間集計、税務申告を代替しない。売上計上時期、勘定科目、経理方式、貸倒要件、申告値は税理士と承認した会社方針を正とする。

## 9. 状態と監査

- ファイル作成は、修了証明書の交付、DIPS修了者情報連携、本人のDIPS更新申請、見積・請求帳票の送付を意味しない。請求発行は正式会計側ですでに確定しており、帳票作成で発行取引を増やさない。
- 作成確定で必要となる番号・日付・正式請求参照は、Drive作成前にサーバーが共有正本へ予約保存する。ブラウザから作成後に書き戻さず、業務状態も自動確定しない。
- 生成台帳には作成日時、対象レコードID、成果物種別、入力ハッシュ、版、結果、ファイルID・URL、実行者、採番情報を記録する。
- 生成台帳はDrive書込前に `prepared` を永続化するWALとして使い、正常完了は同じ予約行を `created`、確実な失敗は `error` へ読戻し検証付きで更新する。`prepared` が残る再実行はDriveを先に再照合し、結果不明時に新しい版・重複ファイルを作らない。
- 同一内容の再実行は二重作成せず、既存ファイルを返す。正式請求帳票の同一性は対象請求の不変キー、発行時スナップショット、明細で判定し、別顧客等の無関係な会計イベントで変わる会計全体ハッシュは版判定へ含めない。全額取消済み請求の通常帳票は新規生成せず、既存成果物の履歴参照だけを許可する。
- 入力変更後の再実行は旧版を削除・上書きせず、訂正版を作る。
- 証明書有効期限の基準は初回発行日とし、訂正・再発行日で延長しない。現版は再発行日の別管理に未対応のため、初回正式成果物作成後の発行日・期限変更は停止し、担当部署確認を求める。
- 共有正本の監査ログは追記型とし、操作時刻、操作者、対象キーのハッシュ、操作、理由コード、承認者、変更前後ハッシュ、前後版、相関IDを保存する。対象者の氏名・連絡先・本文を監査ログへ重複保存しない。
- 通常の対象者更新・無効化・ロール変更では、利用者入力の承認者を監査証跡として受け入れない。承認者欄は、サーバーが別担当者本人を確認した二者承認フローだけで記録する。
- 1件更新は `PREPARED` の監査記録、データ書込、`COMMITTED` の順で行う。正本行保存後に監査確定だけが失敗した場合は、未保存とは表示せず `committed: true`、`recoveryNeeded: true`、監査相関IDを返し、画面は同じ内容を再登録せず監査だけを再照合する。次のロック更新前にも保留マーカーを確認し、正本と一致する場合だけ `COMMITTED` を補完する。未適用のPREPAREDと別内容が同じ対象で競合する場合は上書きを停止する。
- 会計は各コマンドごとに連続版、前状態ハッシュ、前イベントハッシュ、現状態ハッシュ、冪等キーを保存し、チェーン欠落・改ざん・投影不整合を検出した場合は停止する。派生物だけの欠落・不整合は検証済みイベントから再構築する。

## 10. 閲覧と個人情報

一覧は判定、対応期限、対象、必要な対応、期限の根拠、状態を優先する。免許番号、電話、メール、会社名、修了証明書番号、身体適性確認結果、内部メモは詳細画面で必要時だけ表示する。

## 11. 本番データ管理

取込、保存、閲覧、請求、消費税、入金消込、集計、権限、監査ログは `OPERATIONS_DESIGN.md` を正とする。参照スプレッドシートと画面・成果物項目の対応は `SOURCE_FIELD_MAPPING.md` を正とする。

確認済みのGoogle版は現在も `Authorization needed` で停止している。所有者認可、Workspace移管と実ID、実Drive保存・権限、印刷・PDF、対象者世代復元、会計災害復旧とポインタ切替前後の障害を含む結合試験は未完了であり、完了するまで実データの正本・正式帳簿として本番利用しない。

共有正本は明示的な物理削除を提供せず、バックアップの標準保持月数を36か月としている。ただし日次バックアップトリガー、定期検証、廃棄承認、実際の3年間保持は自動的には開始されず、実環境の運用設定と定期試験が必要である。`localStorage` はこの保存要件を満たす正本ではない。

手動・日次バックアップは、1つの安定した冪等キーとシステム実行IDで、対象者・ロールのStoreバックアップとFinanceバックアップを対にする。両方が同じ取得時点のStore fingerprint、Finance revision、Finance state hashに一致した `COMPLETE` だけを成功とする。StoreバックアップJSONは、作成直後、冪等再利用、読込、復元時にファイル単位のPRIVATE設定と直接共同利用者がいないことを検査する。`PARTIAL` は対の片方まで作成された未完了状態、`ABORTED_PARTIAL` は取得元の変化等により旧試行を証拠として固定した状態で、いずれも復旧元の完全対として扱わない。同じ冪等キー・同じ内容の再送は同じ完了結果を返し、同じキーの異なる用途は拒否する。

共有正本の初期設定時点で、対象者とロールをハッシュ付きの初回世代へ確定し、検証済み `activeDataGeneration` を正本にする。通常の単票更新は、監査 `PREPARED` と正本行の間で応答を失っても同じ更新を推測再実行しない。V3保留証明に、相関ID、更新前世代ハッシュ、作成・更新時刻だけを正規化した非個人情報の行証明を保存し、再送時は元のライフサイクル時刻を復元して行全体と世代集計が完全一致した場合だけ既存更新として回収する。旧V2以前の `BEFORE` 保留は厳密な時刻証明がないため、自動再送せず管理者確認で停止する。

対象者・ロール復元は、Apps Script版の管理者画面から、申請管理者が登録済みバックアップ、理由、別の管理者を指定してdry-runし、その指定を受けた管理者本人だけが15分以内に、保留一覧の正確な `batchId` と明示確認を再入力して確定または却下できる。確定前に共有正本の状態を再照合し、復元前安全バックアップを作る。バックアップ内の有効ロールへ復元時点の同一Workspace・個人所有者制約を再適用する。対象者とロールは2枚の新しい世代シートへ全件書込・検証し、単一の `activeDataGeneration` ポインタでまとめて切り替える。正本との差分0件はバックアップも新世代も作らない無操作とする。準備途中で停止した世代を正本として読まず、旧世代を削除しない。旧世代と未完了世代は16件から警告し、24件で新しい一括取込・対象者復元を安全停止する。全シートの割当セル数も実測し、700万セルから警告する。新世代2枚、世代・バックアップ台帳、初回ポインタ、未完了バックアップ再調整と全PREPARED／COMMITTED監査行を予約して800万セル、または新世代・台帳20万行を超える場合は最初の書込前に停止する。通常の単票更新は継続し、自動削除・自動圧縮は行わない。公開Pages版では復元できず、実Driveでの強制終了・復元試験は未実施である。

会計災害復旧は、登録済みFinanceバックアップのうち最新の `COMPLETE` だけを候補とする。バックアップ後の会計イベント行がある場合、または現台帳が正常でバックアップと異なる場合は、日常的な過去版への巻戻しとして拒否する。申請管理者と指定された別管理者、15分期限、正確な依頼ID・固定確認文を必須とし、元Storeを直接上書きしない。専用データフォルダへStore全体の非公開ステージコピーを作り、会計イベント、チェックポイント、監査、承認、投影、対象者参照、請求成果物参照、管理者2名を検証し、復旧後ベースラインバックアップを作成・検証してからStoreポインタを1回切り替える。旧Store IDは安全退避先として保持する。コード実装済みだが、実Driveでの障害・応答喪失・権限試験は未実施である。

旧FinanceStore V1の `finance_snapshots` はV2として開かず、V1からV2への移行は旧台帳の凍結、新規V2台帳、二者承認、全件照合、切戻し手順を持つ独立した承認済み処理として別途実装・試験する。

メール送信、DIPS連携、通知・バックアップの定期トリガーを本番有効化する場合は、接続先、読取・書込範囲、権限、操作ログ、バックアップ、デプロイ前レビューを別途承認する。

## 12. 実装モジュールと責任範囲

| ファイル | 責任 |
|---|---|
| `Code.js` | Web公開APIとエラー整形。保存・会計・成果物の実処理を持たない |
| `DataStore.js` | 専用共有正本、ロール・Workspace所有者制約、楽観ロック、監査、旧ブラウザ移行・削除確認、Store/Financeシステムバックアップ対、対象者・ロールの二者復元 |

| `FinanceStore.js` | V2追記型会計イベント、分割チェックポイント、冪等性、会計監査、二者承認、投影修復、全再生健全性検査、`PREPARED`→`COMPLETE`会計バックアップ |
| `FinanceDisasterRestore.js` | 最新 `COMPLETE` 限定の二管理者災害復旧、Store全体ステージ、復旧後ベースライン、ポインタ切替 |
| `SourceImport.js` | 更新予定者一覧の読取専用preview/commitと列・ハッシュ検査 |
| `Artifacts.js` | 正本ID・版・ハッシュ検査、設定、原本保護、採番dry-run、Drive前予約、発行済み正式請求の帳票化、成果物生成のオーケストレーション |

| `ArtifactCalendar.js` | 公式祝日CSV、DIPS追加閉庁日、5営業日暫定計算の検証 |
| `Index.html` | 画面、入力検証、Apps Script API呼出、公開Pages版のメモリ上合成サンプル閲覧、同一オリジン旧保存の明示救出 |
