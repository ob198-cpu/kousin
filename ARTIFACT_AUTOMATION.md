# 更新講習 成果物自動作成 運用仕様

確認日: 2026-07-24

## 1. 対象範囲

専用共有正本へ保存済みの対象者レコードから、利用者が選択した次の6種類をDriveへ作成する。

1. 修了証明書発行台帳
2. 更新講習修了証明書
3. DIPS提出CSV
4. 更新講習案内
5. 講習記録簿
6. 見積書・請求書

確認済みのGoogle版は現在も `Authorization needed` で停止している。デプロイ所有者が認可を完了し、実Driveの保存・権限、再実行、訂正版、部分失敗、印刷・PDFをテストデータで結合試験するまでは、本番成果物の作成・交付・送付に使用しない。

対象者の保存と成果物作成は別操作にする。保存時や画面表示時に自動作成せず、「作成前検査」へ合格した後の明示確定だけでDriveへ書き込む。公開APIはブラウザ内の対象者本文を正本として受け付けず、`recordId`、画面で確認した `expectedVersion`、`expectedPayloadHash` を必須とする。サーバーは作成前とロック取得後に共有正本を再読込し、版またはハッシュが変わっていれば停止して最新正本を返す。

ファイル作成は、証明書交付、講習機関によるDIPS修了者情報連携、受講者本人のDIPS更新申請、見積送付、売上計上、入金消込を自動確定しない。請求帳票だけは、正式会計ですでに発行済みの請求を印字するものであり、ファイル作成によって新たな請求発行取引を作らない。

原本ファイルには書き込まない。テンプレートを差し替える場合は、次を先に確認する。

- アプリ実行者に原本の読取権限があること
- 表名、タブID、表の行列、置換対象文字が現行仕様と一致すること
- 見本の氏名、日付、料金、事業者情報が出力へ残らないこと
- コピー作成の試験後も原本の更新日時・内容が変わっていないこと

更新講習案内は単一の想定タブ、空の対象者差込欄、日程表構造を検査する。講習記録簿は一等・二等の2シートだけであること、氏名・日付・会場等の差込セルが空であること、必要な標題・最低時間セルが一致することを検査する。想定外タブ、シート、実データ、差込構造の変更があれば作成を停止する。原本編集者を限定し、変更は承認・試作出力・個人情報検査を経て反映する。

更新講習案内と講習記録簿は、全本文・全使用セルを目視確認した承認版のDrive最終更新時刻もコードへ固定する。原本が更新された場合は、部分的な差込構造検査に合格しても作成を停止する。変更内容、ヘッダー・フッター・脚注・全セルの個人情報不存在、試作出力を再確認し、新しい承認時刻をコードレビューで更新するまで使用しない。

Googleドキュメントの6タブと6成果物の対応、発行台帳参照元、修了証明書参照元は、承認時のDrive revision ID・更新時刻をコードへ固定する。実行時にマニュアル本文を読み直して仕様を自動生成せず、参照元変更を検知した場合は停止する。変更内容、項目対応、計算式、清浄性、試作PDFを再承認してコードの固定値を更新するまで自動追従しない。

機械検査は所定タブ・セル・本文構造を確認するもので、ヘッダー、フッター、注釈、画像等を含む全領域の個人情報不存在を保証しない。専用原本は管理者が全ページを目視確認し、編集者を限定し、試作出力を確認した承認版だけを登録する。

修了証明書と発行台帳は、参照元の直接コピーを運用原本にしてはならない。「専用原本を自動準備」は、証明書の一時コピーから不要タブを削除し、番号・氏名・日付・申請番号・講師・機関情報・区分選択を固定ダミー値または空欄へ清浄化した後、その状態を二次コピーする。最終原本には清浄化後の版だけが入り、一時コピーは完全削除する。完全削除とゴミ箱移動の両方に失敗した場合は停止してIDを表示する。台帳は参照元ファイル全体を複製せず、空の`ベース`シートだけを新規ブックへコピーする。無個人情報の専用原本IDが事業者設定へ登録され、証明書は単一ベースタブ、台帳は空の`ベース`だけであることを検査できるまで作成を停止する。

証明書のベース本文にも実在者情報を残さない。専用原本は、技能証明申請者番号=`0000000000`、講習日=`2000年1月1日`、期限=`2000年4月1日`、氏名空欄、担当講師空欄という検査用センチネルへ置換する。台帳専用原本は`ベース`以外のシートを持たず、データ行をすべて空欄にする。センチネルや空欄条件が崩れた原本は作成を停止する。

## 3. 出力構成

成果物の親フォルダは、承認済みのDriveフォルダ [2026年度](https://drive.google.com/drive/folders/1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN)（ID `1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN`）へ固定する。利用者入力、保存済み設定、API要求のいずれからも別IDへ変更できない。対象は2026年度（2026-04-01から2027-03-31）とし、対象者年度または講習修了日の年度が異なる場合は停止する。初回の作成確定時に、システムが固定識別マーカーを付けた `自動作成` フォルダを作る。2027年度以降は年度名だけから推測せず、新しい承認済み親ID、固定年度、権限、結合試験をコード変更として承認する。

新規フォルダ、Googleドキュメント／スプレッドシートのコピー、CSVは、Drive APIの作成・コピー要求に親フォルダを含め、Workspaceの既定公開を無効にして作成する。作成直後に「制限付き」、親が1か所、許可メールとの完全一致、domain/anyoneなし、編集者再共有なしを確認し、合格する前に対象者情報を書き込まない。CSVは空ファイルのACL検査後に本文をアップロードする。専用原本フォルダと専用原本ファイルは所有者だけの権限とし、同じ全権限検査を行う。

```text
2026年度（1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN）
└─ 自動作成
   ├─ 更新講習_成果物レジストリ
   ├─ 更新講習修了証明書発行台帳_YYYY年
   └─ 対象レコード別フォルダ
      ├─ 講習修了証明書_対象者_vN
      ├─ DIPS更新修了者_対象者_vN.csv
      ├─ 更新講習のご案内_対象者_vN
      ├─ 講習記録簿_対象者_vN
      └─ 見積書・請求書_対象者_vN
```

2026-07-24の接続確認時点で固定親フォルダは空、制限付き、所有者のみだった。これは将来の安全性を保証するものではないため、設定保存時、作成前検査時、作成確定時に毎回 `DriveApp.Access.PRIVATE` であることを確認し、リンク公開・ドメイン公開のフォルダには作成しない。ただし `PRIVATE` は個別共有がないことを意味しない。Drive APIの権限一覧を全ページ取得し、`user` / `group` は承認済みメール一覧と完全一致させ、`domain` / `anyone`、メール不明、許可外権限、ページ取得未完は停止する。Googleグループを許可する場合はグループ構成員を別途監査し、各操作担当者には本人メールの直接 `user` 権限も付与して、本人メールとグループメールの両方を許可一覧へ記載する。出力親、生成フォルダ、台帳、成果物は編集者による再共有を禁止し、確認不能または許可状態なら停止する。テンプレート原本の共有設定を出力運用の権限設計とみなさない。

新規の成果物レジストリ、年次発行台帳、見積書・請求書は、Drive APIの作成要求へ親フォルダIDを同時指定して固定先へ直接作る。My Drive直下へ一度作ってから移動する方式を使わず、作成と移動の間で中断して固定先外へ孤児ファイルが残る経路を作らない。

`自動作成` フォルダと対象レコード別フォルダには、親フォルダID・対象レコードIDを含む固定識別マーカーを説明欄へ保存する。保存済みIDまたは同名候補を再利用するときは、名前、親、共有範囲、許可メール、識別マーカーがすべて一致する場合だけ使用し、Script Properties消失時も同名の別用途フォルダを推測で流用しない。各Drive作成・コピーの要求前に、操作キーと `ATTEMPT_STARTED` をScript Propertyへ保存・読戻す。要求結果が不明なら `OUTCOME_UNCERTAIN`、ID・親・種類・権限まで検証済みだが上位の設定・生成台帳へ未確定なら `CREATED_VERIFIED` を残す。成果物台帳、設定IDまたは年次台帳IDへ同じIDを確定して読戻した後だけ解除し、固定先内の一覧が0件でも未作成とは決めず、正確な実体を照合できるまで同じ操作を再送しない。

台帳以外の個別成果物には `recordId`、成果物種別、成果物入力ハッシュ、成果物版、出力内容ハッシュを含む固定識別マーカーを保存する。生成台帳の補助情報には、Drive作成前に確定した共有正本の版と `payloadHash` も保存する。再利用時は、レジストリのfileId・実ファイル名・URL・親フォルダ・識別マーカー・出力内容ハッシュ・共有権限が実ファイルと完全一致する場合だけ既存成果物を返す。年次発行台帳はファイル識別に加え、対象行のrecordId・成果物版・入力ハッシュ・可視値ハッシュを照合する。

`自動作成`、対象者別フォルダ、生成台帳を手作業で先に作成、移動、改名しない。識別マーカーのない同名項目は自動認定せず停止する。固定化前の旧出力先設定、旧フォルダ、旧レジストリ、旧年次台帳、`prepared`・`error` 履歴は自動移動・自動削除しない。旧フォルダ直下が空で、自動作成登録履歴と未解決の削除失敗記録がないことを保存時に機械確認できた場合だけ固定先へ切り替える。残存項目、履歴、確認不能がある場合は固定親への新規作成も停止し、管理者が内容・親フォルダ・共有者・採番・全履歴を照合して、承認された監査付き移行または別名退避を完了した後に切り替える。名前や見出しが似ているだけの既存項目へ説明欄を後付けして移行済み扱いにしない。

成果物設定、振込先、設定版、更新者、変更hash、旧保存先履歴は単一のV2 Script Propertyへ保存し、書込後に同じJSONとhashを読戻す。通常設定保存と案内日程保存は読込時の `settingsVersion` とidempotency keyを必須とし、競合時は再読込まで停止する。応答だけ失われて書込済みの場合は読戻し一致で成功扱いにし、V2破損時はV1へfallbackしない。V2を完全検証した後だけ旧V1設定と旧平文振込先を削除し、全propertyの読戻しでV2同一・旧2キー不存在を確認する。清掃結果を確認できない場合はV2を維持して次の設定更新と専用原本準備を停止する。確定した変更hash・版・更新者は共有正本のCOMMITTED監査行へ同じcorrelation IDで記録し、監査未回収中は次の設定変更を停止する。振込先本文は画面応答・変更履歴・監査行へ出さない。現行バックアップには監査hashは入るが設定本文を含まないため、設定災害復元は別途整備が必要である。

固定親の対象は成果物、生成台帳、年次発行台帳だけである。対象者・ロールの共有正本、FinanceStore、バックアップ、無個人情報の専用原本は、それぞれ専用の非公開領域に分離する。

## 4. 事業者設定

成果物作成前に次を設定する。

| 設定 | 使用先 | 取扱い |
|---|---|---|
| 事業者名 | 証明書、案内、見積・請求 | 設定値を出力する |
| 郵便番号・住所 | 案内、見積・請求 | 正式表記を確認する |
| 電話 | 案内、見積・請求 | 申込先・発行者連絡先として使用する |
| FAX | 見積・請求 | 任意 |
| 申込先メール | 案内 | 形式を検査する |
| 適格請求書発行事業者登録番号 | 見積・請求 | 登録状況と表記を経理担当者が確認する |
| 振込先 | 請求書 | 空欄の再保存では既存値を維持し、画面への本文再表示はしない |
| 固定出力先フォルダ | 全成果物、生成台帳 | `2026年度`（ID `1XmQirjBrQR-uC_GuBVXAyRK5zfqtoQwN`）固定。画面は読取専用で、別IDを拒否する |
| 出力許可メール一覧 | 全成果物、生成台帳 | 必須。Drive APIのuser/group権限を全件照合し、domain/anyoneと編集者による再共有を拒否する |
| 無個人情報の修了証明書原本ID | 修了証明書 | 必須。実名タブのある参照元IDは拒否する |
| 無個人情報の発行台帳原本ID | 発行台帳 | 必須。既存データ入り参照元IDは拒否する |

証明書・発行台帳の専用原本IDが未設定の場合は、事業者設定の「専用原本を自動準備」で一度だけ作成する。保存済みIDがある場合は清浄性検査だけを行い、重複作成しない。専用原本フォルダはMy Drive直下の制限付きフォルダとし、所有者以外との共有や編集者による再共有があれば自動準備を停止する。

採番・DIPS・案内日程は次を共有設定として管理する。

| 設定 | 使用先 | 取扱い |
|---|---|---|
| 採番正本への切替確認 | 証明書番号、DIPS申請者ID | 既存台帳を照合し、必要な最新番号を登録した後だけ有効にする |
| 自動採番の切替年月 | 証明書番号、DIPS申請者ID | 必須。証明書は発行月、DIPS申請者IDは講習修了月で判定し、切替前は確認済み番号を手入力する |
| 既存の最新証明書番号 | 証明書番号 | 任意。対象年月と一致する場合だけ次番号の下限に使う |
| 既存の最新DIPS申請者ID | DIPS申請者ID | 任意。対象年月と一致する場合だけ次番号の下限に使う |
| DIPS追加閉庁日 | DIPS修了者情報連携 | 土日・内閣府公表祝日以外に担当部署へ確認した日だけ登録する |
| DIPS営業日暦の確認日・確認者 | DIPS修了者情報連携 | DIPS CSV作成時は必須。起算日・閉庁日の扱いを担当部署へ確認した記録 |
| 案内日程4行 | 更新講習案内 | Script Propertiesの共有設定だけを正本とし、通常処理ではブラウザへ保存しない |

DIPSのログインID、パスワード、認証情報は保存・使用しない。

## 5. 作成前検査

作成前検査ではDriveへ書き込まず、番号も予約しない。検査合格後、利用者が作成を確定する。

### 共通

- 公開APIの `recordId`、`expectedVersion`、`expectedPayloadHash` が、削除されていない共有正本の最新行と完全一致すること
- ブラウザから対象者本文が送られた場合も、共有正本と完全一致しない本文は拒否し、ブラウザ値を優先しないこと
- 対象者名があること
- 固定保存先IDが承認値と一致し、旧出力先の監査が未解決でなく、アクセス可能かつ非公開であること
- 講習修了後の正式成果物は、技能証明申請者番号、氏名、住所、生年月日、DIPS通知メールの対象講習について、照合結果、照合日、照合者、証憑参照が記録されていること
- 選択した成果物のテンプレートを読み取れること

### 成果物別

| 種別 | 主な必須検査 |
|---|---|
| 発行台帳 | 講習修了日、発行日、交付状態と実交付日の整合、資格区分、証明書番号の形式 |
| 修了証明書 | 講習修了日、発行日、資格区分、技能証明申請者番号、担当講師、発行者情報 |
| DIPS提出CSV | CDP受講分、講習修了日、証明書初回発行日、資格区分、技能証明申請者番号、DIPS用コード値、営業日暦の確認記録 |
| 更新講習案内 | 現免許期限、講習実施可能日、講習締切日、料金、連絡先、4行の日程 |
| 講習記録簿 | 航空機の種類が回転翼航空機（マルチローター）であること、学科・実地会場、各項目の日付・開始・終了・担当者、講習記録日が未来日でなく講習修了日以前であること、最低時間 |
| 見積書・請求書 | 対象者 `recordId` に一致する発行済み正式請求の `financeInvoiceId`、不変キー、会計状態、正式明細、税区分、端数方式、取引日、合計、請求先、発行者、振込先 |

エラーは作成を止める。警告は内容を利用者に示し、DIPS公式ひな形との未照合など、作成後にも人の確認が必要な事項を残す。

## 6. 採番と日付

採番、共有正本への予約、Drive作成、生成台帳更新は `LockService` で直列化する。証明書番号、DIPS申請者ID、見積書番号、計算日を新たに決めた場合は、対象者の期待版を検査したサーバーがDrive作成前に共有正本へ予約保存する。請求書番号と請求日等は発行済み正式請求の値をそのまま使い、成果物側で再採番・再計算しない。

同じ `recordId` の過去作成履歴に証明書番号、DIPS申請者ID、見積書番号、請求書番号がある場合は、その全履歴を照合する。履歴内で番号が競合する場合、または今回入力が履歴の確定番号と異なる場合は作成を停止する。現版は番号取消・番号訂正の専用ワークフローを持たないため、別番号の新版として処理しない。

既存の証明書番号・DIPS申請者IDを照合していない状態では空欄から自動採番しない。運用開始時に既存台帳を確認し、自動採番の切替年月と、切替月に発行済み番号があればその最新番号をseedへ登録してから「採番正本へ切替済み」を有効にする。証明書番号は発行月、DIPS申請者IDは講習修了月を基準に切替とseedを判定する。切替前は既存台帳で確認した番号を手入力し、切替後は生成台帳の最大値とseedの大きい方から次番号を決める。

| 番号 | 形式 | 規則 |
|---|---|---|
| 更新講習修了証明書番号 | `UC0157YYMMNNNN` | 修了証明書発行年月＋同一発行月内4桁連番 |
| DIPS申請者ID | `YYMMNN` | 講習修了年月＋月内2桁連番 |
| 見積書番号 | `QT-UC0157-yyyyMMdd-N` | 更新講習専用名前空間、作成日単位の連番 |
| 請求書番号 | 原則 `INV-UC0157-yyyyMMdd-N` | 正式会計の発行時に確定した番号を使用。成果物作成では変更・再採番しない |

既存CDP管理システムの `QT-yyyyMMdd-N` / `INV-yyyyMMdd-N` と別名前空間にすることで、独立したロックの間で同日番号が衝突しないようにする。既存CDP側ですでに正式採番された番号を手入力して帳票化する場合だけ、従来形式も移行互換として受け入れる。

修了証明書有効期限は次で計算する。

```text
修了証明書有効期限 = 修了証明書初回発行日の3暦月後の前日
```

3暦月後に同じ日が存在しない場合は、その月の末日を期限とする。例: 1月31日発行は4月30日まで、6月10日発行は9月9日まで。

見積日と見積有効期限は共有正本の明示入力を優先し、空欄時の既定値はDrive作成前にサーバーが共有正本へ予約保存する。請求日、取引日、支払期限、請求書番号は選択した発行済み正式請求の値を使用する。ブラウザが作成結果を対象者正本へ後から書き戻す処理は行わない。

修了証明書の有効期限に使う発行日は初回発行日である。訂正・再発行日で初回発行日を変更して期限を延長しない。現版は再発行日の別管理に未対応のため、初回の正式成果物作成後に発行日・期限を変更する操作は停止し、担当部署へ確認する。

DIPSの5営業日は、講習機関が行う修了者情報連携の期限であり、受講者本人の更新申請期間とは別である。公開資料だけでは起算日・閉庁日の取扱いを確定できないため、画面は初日不算入として発行日の翌営業日を1日目に暫定計算する。土日、内閣府公表祝日、設定済み追加閉庁日を除き、未収録年・確認日/確認者なしでは期限を推測せず作成を停止する。必ず「担当部署に確認が必要」と表示する。

## 7. 成果物別の処理

### 7.1 修了証明書発行台帳

- 修了証明書初回発行日の年ごとに年次台帳コピーを1ファイル作る。
- `ベース`シートを年度名へ変更し、B:Iへ証明書番号、対象者、区分、講習日、交付有無、交付日、有効期限、備考を記入する。
- 対象レコードID、版、入力ハッシュは管理列へ保持し、通常表示では隠す。
- 訂正は旧行を削除せず、新版の行を追加する。実際に交付した版は別途確認記録で特定する。

台帳ファイルへの追記は成果物の生成記録であり、証明書の交付済み判定ではない。交付有無・交付日は実際の交付事実に基づいて入力する。

### 7.2 更新講習修了証明書

- 証明書テンプレートの `t.0` をコピーへ差し込む。
- 証明書番号、講習修了日、有効期限、対象者名、技能証明申請者番号、一等・二等、担当講師、登録更新講習機関名、機関コードを記入する。
- 無個人情報の単一ベース原本だけをコピーする。ほかのタブがある原本はコピー前に拒否する。
- 電子印影は自動で付けない。押印・承認・交付は作成後に確認する。

### 7.3 DIPS提出CSV

- `SOURCE_FIELD_MAPPING.md` の11列を、ヘッダー1行・データ1行、UTF-8 BOM付き、CRLFで出力する。
- 登録更新講習機関コードは `0157`、事務所コードは `R0157001` を使う。
- 身体適性検査証明書番号は現行ガイドライン指定の固定値 `PA000000000000` とし、内部管理用の身体適性資料番号を転記しない。
- 状態フラグは新規=`1`、訂正=`2`、削除=`3` とする。訂正・削除は誤登録時と同じ申請者IDを必須とし、自動採番しない。
- 自動ログイン、自動アップロード、自動申請は行わない。
- DIPSの最新公式CSVひな形とは未照合のため、アップロード前に担当者が列名、順序、桁数、コード値を照合する。
- CDP受講分だけを作成対象とする。他機関受講分をCDPが連携できる正式根拠は未確認のため作成を停止する。
- CSVファイルの作成日、機関による実際のアップロード受理日、本人の更新申請日は別項目とする。CSV作成だけで連携済みにしない。
- 修了証明書発行後5営業日の暫定期限と実連携日を比較し、超過は警告・遅延理由・担当部署確認の対象にする。

### 7.4 更新講習案内

- 対象者名、現免許期限、講習実施可能期間、一等・二等、税込料金、申込先を記入する。
- 日程マスタは4行すべてに日付、会場、午前・午後・夜間のうち1つ以上を必要とする。
- 各日程が対象者の講習実施可能日から講習締切日までに入ることを検査する。
- テンプレートの見本日付、見本会場、見本料金を流用しない。
- 案内文書を作ってもメール送信や案内送付済みへの更新は行わない。

### 7.5 講習記録簿

- 現在の自動作成と最低時間判定は、回転翼航空機（マルチローター）に限定する。ヘリコプター・飛行機・未確認では作成を止める。
- 共通学科（一等は一等追加項目を含む）は全受講者で必須とする。実地の操縦演習・指導及び質疑応答は、技能証明の効力停止処分者向け講習が「あり」の場合だけ必須とし、「未確認」では正式成果物を作成しない。
- 資格区分に応じて「一等無人航空機操縦士」または「二等無人航空機操縦士」を選ぶ。
- 出力コピーは一等をA1:H32、二等をA1:F32へ整え、不要なシート・行・列を除く。
- スプレッドシートのタイムゾーンを `Asia/Tokyo` に設定し、業務日付を文字列として記入する。
- 受講者、区分、講習修了日、学科会場、実地会場、各講習項目の日付・時刻・担当者を記入する。

最低時間は次を検査する。

| 区分 | 項目 | 最低時間 |
|---|---|---:|
| 共通学科 | 5項目合計 | 30分 |
| 共通学科 | 動画 | 20分 |
| 一等学科 | 一等留意事項 | 15分 |
| 一等学科 | 一等動画 | 10分 |
| 一等実地（停止処分者のみ） | 操縦演習 | 5分 |
| 二等実地（停止処分者のみ） | 操縦演習 | 6分 |
| 二等実地（停止処分者のみ） | 指導・質疑応答 | 5分 |
| 一等実地（停止処分者のみ） | 指導・質疑応答 | 10分 |

終了時刻は開始時刻より後とし、日をまたぐ時間入力は認めない。同じ日の講習項目は半開区間 `[開始, 終了)` として重複を検査し、終了時刻と次の開始時刻が同じ連続枠は許可する。

### 7.6 見積書・請求書

- 公開APIは `financeInvoiceId` を必須とし、FinanceStore V2から請求を再読込する。
- 対象者 `recordId` と請求の `customerId` が一致し、状態が `ISSUED`、不変キーが存在し、会計台帳が復旧待ちでなく、有効請求残額が0より大きい場合だけ新規作成する。下書き、全額取消済み、別対象者の請求は拒否する。取消前に作成済みだった成果物は履歴として参照できるが、取消表示のない通常請求書を新しい版として生成しない。取消の反対取引で有効残額が正式に復元された場合だけ再検査する。
- 対象者レコード内の旧 `feeExTax`、`discountExTax`、`taxRate`、`invoiceStatus` は帳票金額の正本として使用しない。
- V3レイアウトは、税抜入力 `EXCLUSIVE`、単一税区分、課税10%または課税8%、最大10明細だけを正確に表現する。税込単価、複数税率、非課税・不課税・対象外、11行以上など、現レイアウトで正確に表現できない請求は停止する。
- 正式請求の各明細ID、摘要、数量、税抜単価、請求/値引区分、税区分、明細金額をそのまま見積書・請求書の2シートへ記入する。電子印影は自動で付けない。
- 正式請求の請求書番号、請求日、取引年月日（役務提供日）、支払期限、端数方式、税率別合計、請求先と、事業者設定の発行者、登録番号、振込先を記入する。

```text
明細金額   = 数量 × 税抜単価 × (請求明細なら 1、値引明細なら -1)
税抜合計   = Σ(正式請求の明細金額)
消費税     = 正式請求の単一税率グループ税額
税込合計   = 税抜合計 + 消費税
```

作成前に、明細の数量・単価・区分から再計算した税抜合計、税率グループの税抜・税・税込、請求全体の税抜・税・税込がすべて一致することを検査する。帳票側で独自に税額を作り直したり、端数差を調整したりしない。

更新講習の標準画面は課税10%専用である。課税8%を含む正式請求は、FinanceStoreに明細ごとの根拠資料、適用理由、承認者、承認日が保存され、かつV3の単一8%レイアウト検査に合格した場合だけ帳票化できる。非課税・不課税・対象外は現V3で停止する。

見積・請求ファイルは発行済み正式請求の表示物であり、作成だけでは見積送付済、帳票送付済、売上計上済、入金済、消込済にしない。成果物の同一性ハッシュは、対象請求の不変キー、発行時に封印した請求スナップショット、正式明細、帳票条件から作る。別顧客の入金等でも変わる会計全体状態ハッシュは版判定へ含めず、監査時点の補助情報としてだけ記録する。

## 8. 冪等性、訂正、部分結果

生成台帳は次を記録する。

- 作成日時
- 対象レコードID
- Drive作成前に確定した共有正本の版・`payloadHash`
- 成果物種別
- 入力ハッシュ
- 版
- 状態（`prepared`、`created`、`error`。再利用時は既存の `created` 記録を参照）
- ファイルID、URL、ファイル名、出力先フォルダID
- 実行者
- 採番情報
- メッセージ、補助情報、スキーマ版

同じ対象レコード・成果物種別・入力ハッシュで再実行した場合、ファイルが存在し、レジストリと実体の親、名前、URL、識別マーカー、内容ハッシュ、Drive版が一致すれば二重作成せず既存ファイルを返す。入力や設定が変わった場合は版を1つ進めて新規作成する。原本テンプレートを使う成果物は原本IDと最終更新日時、コード生成のCSV・請求帳票はレイアウト版もハッシュへ含めるため、様式改定後は旧成果物を誤って再利用しない。

作成確定では、ロック取得後に正本版・ハッシュを再検査し、採番・計算日・正式請求参照を共有正本へ先に予約保存する。予約後の正本から入力ハッシュを作り、Drive成果物と生成台帳へ結び付ける。Drive作成が失敗しても予約済み番号を別対象へ再利用せず、監査上の予約として残す。APIは予約後の最新正本を返し、画面はそれを再読込する。ブラウザから `recordUpdates` を後書きして正本を変更しない。

Drive書込より前に、生成台帳へ対象レコードID、種別、入力ハッシュ、版、予定保存先、予定ファイル名、採番情報を持つ `prepared` 行を永続化する。この行が成果物作成のwrite-ahead log（WAL）であり、追記結果の応答が不明でも削除しない。正常完了時は同じ行を `created` へ更新し、ファイルID、URL、内容ハッシュ、Drive版、更新時刻を読戻し検証する。

中断後に `prepared` が残っている場合は、新しい版やファイルを先に作らない。予定した固定保存先フォルダ内だけを調べ、親、予定名、予約識別マーカーまたは完成識別マーカー、内容ハッシュ、Drive版が同じ候補を照合する。完成済み候補が正確に1件だけなら同じ予約を `created` へ確定する。候補なしでも、Drive一覧の反映遅延とDrive要求前の中断を一意に判定できないため、新規作成せず「担当部署に確認が必要」で停止する。現在の実行が永続予約を新規作成し、その直後の照合で候補がない場合に限り、同じロック内で最初のDrive作成へ進む。作成途中、候補複数、レジストリ更新結果不明、台帳行の部分書込も停止し、重複作成しない。

成果物と台帳行を安全に巻き戻せたことが確認できる失敗は、同じ `prepared` 行を `error` へ更新し、エラー内容と清掃失敗があれば追跡情報を保存する。`error` となった版も使用済みとして版列に残し、次の作成で同じ版番号を再利用しない。レジストリ更新結果自体が不明な場合は `error` へ決め打ちせず、再照合まで `prepared` を維持する。完全削除または台帳rollbackを確認できない場合は、不透明な対象ID、種別、台帳位置・版、ファイル名等のハッシュ、エラーコード・本文ハッシュだけを持つ `RENEWAL_ARTIFACT_CLEANUP_FAILURE_<ID>` を保存し、1種類だけでなく全成果物・専用原本の新規作成を停止する。氏名を含み得るファイル名、URL、生のエラーや権限メールはScript Propertiesへ保存しない。

清掃事故の復旧では、未解決propertyの本文を証拠として先に退避し、記録されたIDの実体、親、識別マーカー、権限、生成台帳行を照合する。誤作成物の完全削除または承認済み退避、台帳rollback、再読込による不存在・整合確認、復旧理由・実施者・確認者の記録後だけ、該当する1個のpropertyを個別に解決する。接頭辞での一括削除や、一覧0件・同名だけを根拠にした解決は禁止する。

6種類のうち一部が失敗しても、成功済みファイルを削除しない。結果画面に成果物ごとの「作成」「既存利用」「エラー」とリンクを表示し、失敗分だけ内容を直して再実行する。ただし `prepared` の結果不明がある種別は、担当部署でDrive実体と生成台帳を照合するまで再実行しない。

訂正時は次の順で行う。

1. 元レコードまたは事業者設定を訂正する。
2. 同じ成果物を作成前検査する。
3. 新しい版を作成する。
4. 旧版と新版を比較する。
5. 実際に交付・送付・申請・発行した版を業務記録へ残す。

## 9. セキュリティと禁止事項

- DIPSの認証情報をコード、Script Properties、対象者レコード、文書、CSV、生成台帳へ保存しない。
- 参照マニュアルに平文のDIPS認証情報を残さない。発見時は当該認証情報を即時変更し、文書から削除し、共有範囲とアクセス履歴を確認する。
- 原本テンプレートへ直接書き込まない。
- 作成ファイルを「リンクを知っている全員」へ自動公開しない。
- 作成と同時にメール送信、DIPSアップロード、証明書交付済、請求発行済へ変更しない。
- 作成済み旧版やエラー履歴を担当者判断で削除しない。
- `prepared` 行が残る成果物を、生成台帳とDrive実体の再照合なしに別版として作り直さない。
- `localStorage`、CSVバックアップ、生成台帳だけを対象者データの共有正本にしない。
- ブラウザの対象者本文や作成結果を、共有正本の版・ハッシュ検査なしに成果物入力または作成後書戻しへ使わない。
- 対象者レコードの旧請求欄から正式な請求帳票を作らず、発行済みFinanceStore請求だけを使用する。
- 講習記録・帳簿は講習事務終了後3年間保存し、保持期限、廃棄承認、廃棄ログを共有正本側で管理する。

## 10. 作成後チェック

- 対象者、資格区分、日付、番号、料金、振込先が対象レコードと一致する。
- 作成結果の正本 `recordId`・版・`payloadHash` が、作成開始時に確認した対象と一致し、ブラウザから追加書戻しされていない。
- テンプレートの見本データが残っていない。
- 修了証明書番号の年月と有効期限が、修了証明書初回発行日を基準に一致している。
- 講習記録簿の時間と担当者が実施記録と一致する。
- DIPS CSVは最新の公式ひな形と手動照合した。
- DIPS修了者情報連携日と本人のDIPS更新申請日を混同していない。5営業日の起算日・閉庁日を担当部署へ確認した。
- 見積・請求は選択した発行済み正式請求ID・不変キー・発行時スナップショットに一致し、有効請求残額が0より大きく、V3明細、税抜、税、税込がFinanceStoreと完全一致している。会計全体状態ハッシュは監査補助情報であり、成果物版の同一性判定には使わない。
- 出力リンクと版が生成台帳へ記録されている。
- 交付、送付、申請、発行が未実施なら、業務状態も未実施のままである。

## 11. 実装モジュール

成果物の全体制御、正本再読込、設定、原本保護、作成種別への振分けは `Artifacts.js` が担当する。Drive作成前の `prepared` WAL、版列、実ファイル識別、内容ハッシュ、Drive版、中断後の再照合、`created` / `error` 確定は `ArtifactRegistry.js` に分離する。公式祝日CSV、追加閉庁日、DIPS 5営業日の検証は `ArtifactCalendar.js` に分離する。公開APIは `Code.js` が委譲し、画面の入力値や `localStorage` を成果物正本にしない。
